# توثيق نظام البورصة المتكامل لإدارة المخلفات

## 1. رؤية النظام

يهدف نظام البورصة المتكامل إلى تحويل عملية شراء وإدارة المخلفات في تطبيق Delivery Agent Dashboard من عملية ثابتة إلى نظام ديناميكي وذكي يستجيب لتغيرات السوق والعرض والطلب. سيمكن هذا النظام من:

*   **تحديد أسعار عادلة وديناميكية**: تعكس القيمة الحقيقية للمخلفات بناءً على عوامل متعددة (كمية المخزون، جودة المادة، أسعار الموردين، الطلب الصناعي، كفاءة التجميع).
*   **تحسين إدارة المخزون**: توفير رؤية واضحة للمخزون الفعلي في المخازن وتوقع مستوياته.
*   **تحسين كفاءة عمليات التجميع**: توجيه وكلاء التجميع نحو أنواع المخلفات المطلوبة بأسعار أفضل.
*   **اتخاذ قرارات مستنيرة**: توفير بيانات وتحليلات دقيقة للمسؤولين لاتخاذ قرارات الشراء والبيع.
*   **تسهيل التعامل مع الموردين والمشترين**: إدارة عروض الأسعار والصفقات بكفاءة.

## 2. هيكل قاعدة البيانات (Supabase Schema)

سيتم توسيع قاعدة البيانات الحالية في Supabase بالجداول التالية، مع الحفاظ على ترابطها مع جداول `categories` و `subcategories` و `waste_data_admin` الحالية باستخدام معرفات UUID.

```sql
-- جدول المخازن (Warehouses)
CREATE TABLE warehouses (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  location TEXT, -- موقع المخزن (مثال: القاهرة، الإسكندرية)
  capacity DECIMAL(10,2), -- السعة الكلية للمخزن (بالكيلوجرام أو الوحدة)
  current_stock DECIMAL(10,2), -- المخزون الإجمالي الحالي في المخزن
  manager_id UUID, -- معرف المسؤول عن المخزن (إذا كان هناك نظام للمسؤولين)
  contact_phone TEXT,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- جدول مخزون المواد في المخازن (Warehouse Inventory)
CREATE TABLE warehouse_inventory (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  warehouse_id UUID REFERENCES warehouses(id), -- معرف المخزن
  category_id UUID REFERENCES categories(id), -- معرف الفئة الرئيسية للمخلف
  subcategory_id UUID REFERENCES subcategories(id), -- معرف الفئة الفرعية للمخلف
  product_id UUID REFERENCES waste_data_admin(id), -- معرف المنتج المحدد (مثل PET زجاجات)
  quantity DECIMAL(10,2), -- الكمية الموجودة من هذا المنتج في المخزن المحدد
  unit TEXT, -- الوحدة (مثال: كيلوجرام، طن)
  min_level DECIMAL(10,2), -- الحد الأدنى للمخزون الذي يتطلب تنبيهًا
  max_level DECIMAL(10,2), -- الحد الأقصى للمخزون (للتنبيهات أو لإدارة المساحة)
  last_updated TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  PRIMARY KEY (warehouse_id, product_id) -- لضمان فرادة المنتج في كل مخزن
);

-- جدول الموردين (Suppliers) - يمكن أن يكونوا مصانع إعادة تدوير أو جهات شراء كبرى
CREATE TABLE suppliers (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  category_focus TEXT, -- أنواع الفئات التي يتعامل معها المورد (مثل: بلاستيك، معادن)
  rating DECIMAL(2,1), -- تقييم المورد بناءً على الأداء (الجودة، الالتزام)
  delivery_time_days INTEGER, -- متوسط مدة التوصيل بالأيام
  location TEXT, -- موقع المورد
  contact_phone TEXT,
  email TEXT,
  is_active BOOLEAN DEFAULT TRUE,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- جدول عروض الأسعار من الموردين (Supplier Price Offers)
CREATE TABLE supplier_price_offers (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  supplier_id UUID REFERENCES suppliers(id), -- معرف المورد
  category_id UUID REFERENCES categories(id), -- الفئة الرئيسية للمخلف
  subcategory_id UUID REFERENCES subcategories(id), -- الفئة الفرعية للمخلف
  product_id UUID REFERENCES waste_data_admin(id), -- المنتج المحدد
  proposed_price DECIMAL(10,2), -- السعر الذي يقترحه المورد للشراء (لكل وحدة)
  quantity DECIMAL(10,2), -- الكمية التي يرغب المورد بشرائها بهذا السعر
  unit TEXT, -- الوحدة (كيلوجرام، طن)
  valid_from DATE, -- تاريخ بداية صلاحية العرض
  valid_to DATE, -- تاريخ انتهاء صلاحية العرض
  status TEXT CHECK (status IN ('pending', 'active', 'rejected', 'expired', 'accepted')), -- حالة العرض
  notes TEXT,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- جدول بيانات البورصة المتقدمة (Stock Exchange)
-- هذا الجدول سيحتوي على الأسعار الديناميكية لكل منتج (مخلف)
CREATE TABLE stock_exchange (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  category_id UUID REFERENCES categories(id),
  subcategory_id UUID REFERENCES subcategories(id),
  product_id UUID UNIQUE REFERENCES waste_data_admin(id), -- يجب أن يكون فريداً لكل منتج
  base_price DECIMAL(10,2), -- السعر الأساسي (مرجع)
  current_price DECIMAL(10,2), -- السعر الحالي المتأثر بعوامل السوق
  last_manual_price DECIMAL(10,2), -- آخر سعر تم تعيينه يدوياً
  last_manual_price_set_at TIMESTAMP WITH TIME ZONE, -- تاريخ آخر تعيين يدوي
  min_allowed_price DECIMAL(10,2) DEFAULT 0, -- الحد الأدنى المسموح به للسعر
  max_allowed_price DECIMAL(10,2) DEFAULT 9999.99, -- الحد الأقصى المسموح به للسعر
  total_available_stock DECIMAL(10,2), -- إجمالي المخزون المتاح من هذا المنتج عبر كل المخازن
  avg_supplier_buy_price DECIMAL(10,2), -- متوسط أسعار الشراء المقترحة من الموردين
  demand_level TEXT CHECK (demand_level IN ('low', 'normal', 'high', 'critical')), -- مستوى الطلب الحالي على المادة
  supply_level TEXT CHECK (supply_level IN ('low', 'normal', 'high', 'over_supplied')), -- مستوى العرض الحالي للمادة
  price_change_percentage DECIMAL(5,2), -- نسبة التغيير في السعر
  auto_update_enabled BOOLEAN DEFAULT TRUE, -- هل التحديث التلقائي للسعر مفعل؟
  last_price_update TIMESTAMP WITH TIME ZONE DEFAULT NOW(), -- آخر مرة تم فيها تحديث السعر
  next_scheduled_update TIMESTAMP WITH TIME ZONE -- تاريخ التحديث المجدول التالي
);

-- جدول التنبيهات (System Alerts)
CREATE TABLE system_alerts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  alert_type TEXT CHECK (alert_type IN ('stock_low', 'supplier_offer', 'price_change', 'demand_high', 'supply_low', 'warehouse_capacity')),
  severity TEXT CHECK (severity IN ('low', 'medium', 'high', 'critical')),
  message TEXT NOT NULL,
  warehouse_id UUID REFERENCES warehouses(id),
  supplier_id UUID REFERENCES suppliers(id),
  category_id UUID REFERENCES categories(id),
  subcategory_id UUID REFERENCES subcategories(id),
  product_id UUID REFERENCES waste_data_admin(id),
  is_read BOOLEAN DEFAULT FALSE,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  resolved_at TIMESTAMP WITH TIME ZONE
);
```

## 3. الوظائف الأساسية ومراحل التكامل

### 3.1. إدارة المخازن (Phase 1: Foundation)
*   **إنشاء/تعديل/حذف المخازن**: واجهة لإدارة تفاصيل المخازن.
*   **إدارة المخزون داخل المخازن**:
    *   إضافة/تعديل/حذف كميات المنتجات المخزنة.
    *   ربط المخزون بالمنتجات الموجودة في `waste_data_admin`.
*   **تتبع المخزون**: تحديث `current_stock` في جدول `warehouses` و `quantity` في `warehouse_inventory`.
*   **تقارير المخزون**: عرض إجمالي المخزون لكل منتج وعبر المخازن المختلفة.

### 3.2. إدارة الموردين (Phase 1: Foundation)
*   **إنشاء/تعديل/حذف الموردين**: إدارة بيانات المصانع أو الجهات التي تشتري المخلفات.
*   **إدارة عروض الأسعار**:
    *   تسجيل عروض الأسعار المستلمة من الموردين (السعر المقترح، الكمية، فترة الصلاحية).
    *   تغيير حالة العروض (معلق، نشط، مرفوض، منتهٍ، مقبول).
    *   واجهة لعرض العروض والموافقة عليها.

### 3.3. نظام البورصة والتسعير الديناميكي (Phase 2: Core Exchange Logic)

#### الرؤية:
سيكون هذا هو قلب النظام، حيث يتم تحديد سعر شراء المخلفات من وكلاء التجميع (وبالتالي من المنازل/المحلات). يجب أن يعكس هذا السعر عوامل السوق المختلفة لضمان الربحية والقدرة التنافسية.

#### 3.3.1. العوامل المؤثرة على السعر في نظامك:

1.  **العرض (Supply)**:
    *   **إجمالي المخزون المتاح (Total Available Stock)**: من `warehouse_inventory`. كلما زاد المخزون، انخفض السعر.
    *   **معدل التجميع الأسبوعي/اليومي (Collection Rate)**: كمية المخلفات التي يجمعها وكلاء التوصيل بانتظام. زيادة معدل التجميع تزيد العرض وتخفض السعر.
    *   **كفاءة وكلاء التجميع (Agent Efficiency)**: عدد الوكلاء النشطين ومعدل جمعهم.

2.  **الطلب (Demand)**:
    *   **عروض الأسعار النشطة من الموردين (Active Supplier Offers)**: من `supplier_price_offers`. هذه هي مؤشرات الطلب الخارجي. كلما زاد الطلب من الموردين، ارتفع السعر.
    *   **متوسط أسعار الشراء من الموردين (Avg Supplier Buy Price)**: متوسط الأسعار التي يقترحها الموردون حاليًا.
    *   **الطلبات التاريخية من الموردين**: تحليل أنماط الشراء السابقة للموردين.

3.  **الجودة والتصنيف (Quality & Classification)**:
    *   **مستوى النقاء/الفرز**: المخلفات التي تم فرزها جيدًا (من `waste_data_admin` إذا كانت تحتوي على بيانات جودة) تحقق أسعاراً أعلى.
    *   **درجة الرطوبة/التلف**: بيانات يمكن إضافتها عند التجميع أو الفرز الأولي لتؤثر على السعر.

4.  **عوامل السوق واللوجستية**:
    *   **السعر الأساسي (Base Price)**: سعر مرجعي يمكن تعيينه يدوياً لكل منتج.
    *   **تكلفة التجميع والنقل**: تكلفة تشغيل وكلاء التجميع ومركبات النقل (تحسب من نظامك الحالي).
    *   **العوامل الاقتصادية الكلية**: (اختياري) تأثير أسعار النفط، أسعار المعادن العالمية، التضخم (يمكن إدخالها كبيانات خارجية).

#### 3.3.2. معادلة التسعير الديناميكي (Formula)

سيتم تحديث `current_price` في جدول `stock_exchange` باستخدام معادلة تأخذ في الاعتبار هذه العوامل. يمكن أن تكون وظيفة تُشغل بانتظام (مثال: كل ساعة، كل 4 ساعات) أو عند تغير أحد العوامل الرئيسية.

**نموذج معادلة مقترحة (يمكن تعديل الأوزان):**

```typescript
function calculateNewPrice(product: StockExchangeProduct, allInventory: WarehouseInventory[], allOffers: SupplierPriceOffer[]): number {
    const basePrice = product.base_price;
    let newPrice = basePrice;

    // 1. تأثير المخزون (العرض) - كلما زاد المخزون، انخفض السعر
    // تحديد إجمالي المخزون للمنتج المحدد
    const totalInventory = allInventory
        .filter(item => item.product_id === product.product_id)
        .reduce((sum, item) => sum + (item.quantity || 0), 0);
    
    // يمكن تعريف نقاط مرجعية للعرض (Supply Levels)
    let supplyMultiplier = 1.0;
    if (totalInventory < 200) { // نقص حاد في العرض
        supplyMultiplier = 1.15;
        product.supply_level = 'low';
    } else if (totalInventory < 500) { // نقص في العرض
        supplyMultiplier = 1.08;
        product.supply_level = 'normal';
    } else if (totalInventory > 1000) { // زيادة في العرض
        supplyMultiplier = 0.90;
        product.supply_level = 'high';
    } else if (totalInventory > 2000) { // عرض زائد جداً
        supplyMultiplier = 0.80;
        product.supply_level = 'over_supplied';
    } else {
        product.supply_level = 'normal';
    }
    newPrice *= supplyMultiplier;

    // 2. تأثير الطلب (عروض الموردين) - كلما زاد الطلب وارتفعت عروض الموردين، ارتفع سعر الشراء
    const activeOffers = allOffers
        .filter(offer => offer.product_id === product.product_id && offer.status === 'active' && new Date(offer.valid_to) >= new Date());
    
    let avgSupplierPrice = product.avg_supplier_buy_price || basePrice; // استخدام القيمة الحالية كافتراضي
    if (activeOffers.length > 0) {
        avgSupplierPrice = activeOffers.reduce((sum, offer) => sum + (offer.proposed_price || 0), 0) / activeOffers.length;
        product.demand_level = 'high'; // افتراض أن وجود عروض نشطة يعني طلباً عالياً
    } else {
        product.demand_level = 'low';
    }
    
    // دمج تأثير متوسط سعر الموردين (مثال: 70% وزن للسعر الأساسي المعدل، 30% لمتوسط سعر الموردين)
    newPrice = (newPrice * 0.7) + (avgSupplierPrice * 0.3);

    // 3. تطبيق حدود التغيير للحماية من التقلبات الشديدة
    const maxAllowedPrice = product.max_allowed_price || (basePrice * 1.5); // 50% زيادة كحد أقصى
    const minAllowedPrice = product.min_allowed_price || (basePrice * 0.5); // 50% انخفاض كحد أقصى
    
    newPrice = Math.max(minAllowedPrice, Math.min(maxPrice, newPrice));

    // تحديث نسبة التغيير
    product.price_change_percentage = ((newPrice - basePrice) / basePrice) * 100;

    return newPrice;
}
```

#### 3.3.3. آلية التحديث التلقائي:
*   **وظيفة خلفية (Background Job / Edge Function)**: تشغيل دالة (مثال: Supabase Edge Function أو Cron Job على خادم خارجي) بشكل منتظم (مثال: كل ساعة) لجلب البيانات من المخازن وعروض الموردين وتحديث جدول `stock_exchange`.
*   **التحديث اليدوي**: توفير خيار للمسؤولين لتعديل الأسعار يدوياً إذا لزم الأمر.

### 3.4. نظام التنبيهات (Phase 2: Core Exchange Logic)
*   **تنبيهات المخزون**: عند انخفاض كمية منتج معين عن `min_level` في `warehouse_inventory`.
*   **تنبيهات عروض الموردين**: عند وصول عرض سعر جديد بحالة `pending`.
*   **تنبيهات تغير السعر**: عند حدوث تغير كبير (مثال: أكثر من 10%) في `current_price` في `stock_exchange`.
*   **تنبيهات الطلب/العرض**: عند وصول `demand_level` أو `supply_level` إلى `critical` أو `over_supplied`.
*   **واجهة عرض التنبيهات**: دمجها في لوحة تحكم المسؤول، مع إمكانية قراءة/إخفاء التنبيه.

### 3.5. تكامل نظام التجميع (Phase 3: Deep Integration)
*   **تأثير التجميع على المخزون**: عند قيام وكيل بتسليم مخلفات، يتم تحديث `warehouse_inventory` و `current_stock` في `warehouses` بشكل مباشر.
*   **ربط تسعير البورصة بأسعار الشراء من العملاء**: عندما يحدد العميل نوع المخلف وكميته في التطبيق، يتم عرض السعر الحالي من `stock_exchange` له.
*   **آلية دفع النقاط/الجنيهات**: بناءً على السعر الحالي من البورصة، يتم حساب النقاط أو المبلغ المستحق للعميل وإضافته إلى رصيده (تكامل مع نظام الدفع الحالي).

### 3.6. لوحات التحكم والتحليلات (Phase 3: Advanced Features)
*   **لوحة تحكم البورصة**: عرض رسوم بيانية لاتجاهات الأسعار، مستويات العرض والطلب.
*   **تقارير المخزون**: تفاصيل المخزون، معدلات الدوران، المخزون الراكد.
*   **تقارير الموردين**: أداء الموردين، أفضل العروض، متوسط أوقات التسليم.
*   **تحليلات الربحية**: مقارنة أسعار الشراء من العملاء بأسعار البيع للموردين.

### المرحلة 2: تفعيل البورصة ونظام التسعير (3-4 أسابيع)

    *   تعديل عملية تسجيل التجميع في التطبيق (عندما يسجل وكيل تسليماً) ليتم:
        *   تحديث `warehouse_inventory` و `warehouses.current_stock`.
        *   تحريك عملية تحديث سعر البورصة للمنتج المتأثر.
    *   عرض سعر الشراء الحالي للمخلفات من البورصة لوكلاء التجميع والعملاء في تطبيقهم.
*   **نظام الدفع المتكامل**:
    *   حساب النقاط/المبالغ المستحقة للعملاء بناءً على سعر البورصة وقت التجميع.
    *   تكامل هذه الدفعات مع نظام المحافظ الحالي.
*   **تقارير وتحليلات متقدمة**:
    *   لوحات تحكم تفاعلية (باستخدام مكتبات الرسوم البيانية مثل Recharts أو Nivo) لعرض:
        *   اتجاهات أسعار البورصة بمرور الوقت.
        *   معدلات إشغال المخازن.
        *   تحليلات أداء الموردين.
        *   ملخصات الربحية.
*   **تحسينات الأداء**: استخدام Supabase Realtime لإرسال تحديثات الأسعار والتنبيهات في الوقت الفعلي للواجهة الأمامية.

## 5. التحديات المحتملة والحلول

*   **تعقيد التزامن (Concurrency)**: تحديثات متعددة للمخزون أو الأسعار قد تسبب تعارضات.
    *   **الحل**: استخدام معاملات قاعدة البيانات (transactions) لضمان اتساق البيانات.
*   **دقة البيانات**: جودة البيانات المدخلة (خاصة من وكلاء التجميع) تؤثر على دقة الأسعار.
    *   **الحل**: تطبيق آليات تحقق صارمة عند إدخال البيانات، وتدريب الوكلاء، ومراجعة البيانات يدوياً.
*   **أداء خوارزمية التسعير**: إذا كانت المعادلة معقدة أو البيانات ضخمة، قد يؤثر ذلك على الأداء.
    *   **الحل**: تحسين الاستعلامات، استخدام IndexedDB للتخزين المؤقت في الواجهة الأمامية، وتشغيل العمليات الثقيلة كخلفية أو Edge Functions.
*   **إدارة السياسات (RLS)**: يجب إعداد سياسات Row Level Security (RLS) في Supabase بعناية لكل الجداول الجديدة لضمان أمان البيانات.
    *   **الحل**: مراجعة دورية للسياسات وتطبيق مبدأ "أقل امتياز".
*   **تجربة المستخدم**: قد يكون عرض العديد من البيانات معقداً للمستخدم.
    *   **الحل**: تصميم واجهة مستخدم بديهية، واستخدام الرسوم البيانية والجداول سهلة القراءة، وتوفير خيارات تصفية وبحث قوية.

هذه الخطة المتكاملة توفر خارطة طريق واضحة لتطوير نظام بورصة ديناميكي وفعال لتطبيقك، مع الأخذ في الاعتبار خصوصية مجال تجميع المخلفات.
