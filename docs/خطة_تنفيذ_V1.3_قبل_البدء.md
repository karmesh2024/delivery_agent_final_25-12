# خطة تنفيذ V1.3 قبل البدء (Pre-Execution Implementation Plan)

## 1) الهدف

تنفيذ نظام نقاط النادي V1.3 “نسخة أولى مبسطة” وفق المستندات:

- `docs/نظام_النقاط_الجديد_الاقتراح_والتنفيذ.md`
- `docs/نظام_نقاط_النادي_V1.3_المبسطة.md`
- `docs/تحويل_نقاط_المخلفات_إلى_نقاط_النادي_V1.3.md`

بحيث يحقق:

- نقاط النادي = **مكافآت/جوائز** (مؤجلة شهريًا)
- نقاط المخلفات = **قيمة اقتصادية فورية** (كاش/متجر)
- لا تحويل تلقائي ذاتي للمستخدم بين النظامين في V1.3 (إلا “طلب تحويل” Request Only مع موافقة أدمن)
- الراديو: **Session-based** وحساب النقاط **مرة واحدة** عند الإيقاف

---

## 2) هل واجهات التطبيق ستتأثر؟

### القرار التنفيذي

- **لن نعيد بناء الواجهات**؛ سنقوم بـ **الإضافة فوق الموجود** (Additive changes) مع تعديلات نصوص/منطق.

### التغييرات المتوقعة على UI (ملخص)

- **إدارة النقاط**: ستعرض/تدعم `ACTIVATED` وتُبسّط الخيارات إلى V1.3 (EARNED/USED/ACTIVATED).
- **الرصيد**: سنُظهر للمستخدم “🎁 رصيد هذا الشهر (Pending)” و“✅ رصيد متاح (Available)”.
- **تحويل المخلفات للنادي**: بدل تحويل مباشر، سيتم إضافة قسم/صفحة “طلبات التحويل” + موافقة/رفض أدمن.
- **الراديو**: إلغاء منح النقاط كل دقيقة، واستبدالها بمنح واحد عند stop (مع معالجة sessions المتروكة).

> ملاحظة: قد تتغير بعض المصطلحات في الواجهة (UX wording) فقط، بدون تغيير تصميم كبير.

## 3) تحليل الحالة الحالية (As-Is)

- قاعدة البيانات الحالية (حسب `supabase/migrations/20250102_create_club_zone_schema.sql`):

## 5) خطوات التنفيذ البرمجية (من غير كود الآن — فقط خطة)

### المرحلة A — قاعدة البيانات (Supabase Migrations)

1. **Migration: wallet upgrade**
   - إضافة `pending_points`, `available_points`, `used_points` إلى `club_points_wallet`.
   - إبقاء `points_balance` للتوافق مع UI الحالي في المرحلة الأولى:
     - `points_balance` سيصبح “مرآة” للـ `available_points` (نفس القيمة).

2. **Migration: transactions constraint**
   - تعديل constraint في `club_points_transactions` ليقبل فقط:
     - `EARNED`, `ACTIVATED`, `USED`.

3. **Migration: update_club_points_wallet V1.3**
   - تعديل function لتصبح المرجع الوحيد لتعديل المحفظة:
     - `EARNED` → يزيد `pending_points` + `lifetime_points`
     - `ACTIVATED` → ينقل من pending إلى available ويحدث `points_balance`
     - `USED` → يخصم من available ويزيد used ويحدث `points_balance`

4. **Migration: monthly activation**
   - إنشاء `activate_monthly_points` يدويًا + يسجل في `monthly_points_settlement`.

5. **Migration: conversion requests**
   - إنشاء جدول `recycling_conversion_requests`.
   - بدون workflow معقد: فقط `PENDING/APPROVED/REJECTED`.

6. **Migration: radio sessions**
   - إنشاء جدول `radio_sessions`.
   - إنشاء function `close_abandoned_radio_sessions()` لإغلاق sessions المتروكة (مثلًا بعد 4 ساعات).

### المرحلة B — Services/Types/Redux

1. تحديث Types:
   - إضافة حقول wallet الجديدة في `src/domains/club-zone/types/index.ts`.
   - تضييق استخدام transaction types في واجهة النادي إلى V1.3.

2. تحديث `clubPointsService`:
   - وقف/إزالة التحويل المباشر `convertWastePointsToClubPoints` (أو حصره أدمن داخليًا).
   - إضافة:
     - `createConversionRequest`
     - `approveConversionRequest`
     - `rejectConversionRequest`
   - عند الموافقة:
     - خصم نقاط المخلفات اقتصاديًا عبر `points_transactions` (financial-management)
     - منح نقاط النادي عبر `update_club_points_wallet(..., 'EARNED', source='waste_conversion')`

3. تحديث `clubRewardsService`:
   - التحقق من الرصيد يعتمد على `available_points` (أو `points_balance` كمطابقة مؤقتًا).

4. تحديث الراديو:
   - إلغاء منح النقاط كل دقيقة.
   - تنفيذ Session flow:
     - start → create `radio_sessions`
     - stop → update stop_time → calculate minutes → award once

### المرحلة C — UI (إضافة فوق الموجود)

1. `/club-zone/points`
   - تبسيط الفلاتر والـ labels إلى V1.3.
   - إضافة عرض واضح للفروق بين Pending/Available في شاشة منفصلة أو كارت ملخص.

2. `/club-zone/points/conversions` (صفحة جديدة للأدمن)
   - عرض طلبات التحويل + أزرار approve/reject.

3. Banner الاعتماد الشهري
   - Banner في لوحة التحكم أو صفحة النقاط عند وجود شهر لم يتم اعتماده.

4. UX wording
   - Pending = “🎁 رصيد هذا الشهر”
   - Available = “✅ رصيد متاح”
   - نص توضيحي: “النقاط تُعتمد نهاية الشهر للحفاظ على استدامة الجوائز”

## 6) المخاطر (قبل التنفيذ) وكيف سنحمي الإنتاج

### 6.1 مخاطر البيانات

- **تغيير schema للـ wallet** قد يكسر أي query يفترض أعمدة قديمة فقط.
- **تغيير check constraint للمعاملات** قد يفشل inserts قديمة (إن كان يوجد كود ما زال يكتب BONUS/CONVERTED…).

الحماية:
- تنفيذ migrations على branch/بيئة تجريبية أولًا.
- تحديث الخدمات التي تقوم بإدراج معاملات قبل تفعيل constraint.

### 6.2 مخاطر الراديو

- تحويل النظام من “نقاط كل دقيقة” إلى “مرة واحدة” قد يحتاج معالجة جلسات متروكة.

الحماية:
- إضافة `close_abandoned_radio_sessions` وتشغيلها دوريًا.

### 6.3 مخاطر تجربة المستخدم (نفسية)

- سؤال: “ليه مش هقدر أستخدم رصيد هذا الشهر فورًا؟”

الحماية:
- نصوص UX جاهزة + توضيح تاريخ الاعتماد القادم.

- rollback DB migrations: بإزالة الأعمدة الجديدة وإرجاع function القديمة (آخر نسخة مستقرة).
- تعطيل صفحة طلبات التحويل وإرجاع التحويل اليدوي القديم (Admin-only مباشرة) مؤقتًا.

## 8) المخرجات المتوقعة بعد التنفيذ (Definition of Done)

- Wallet يدعم: Pending/Available/Used + UI يعرضها بشكل واضح.
- اعتماد شهري يدوي يعمل ويحوّل Pending → Available.
- تحويل المخلفات: Request Only + approve/reject + تسجيل محاسبي واضح في النظامين.
- الراديو: Session-based + احتساب مرة واحدة + إغلاق sessions المتروكة.
