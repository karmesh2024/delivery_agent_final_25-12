# الفرق بين نظام الصلاحيات العام ونظام الموافقات لإدارة المخلفات

## 📊 نظرة عامة

### نظام الصلاحيات العام (موجود مسبقاً)
**الهدف**: التحقق من **"هل يمكنك القيام بهذا العمل؟"**

### نظام الموافقات (الجديد في migration)
**الهدف**: التحقق من **"هل تمت الموافقة على هذا العمل؟"** + **تسجيل وتتبع العمليات**

---

## 🔍 الفروقات التفصيلية

### 1. **نظام الصلاحيات العام**

#### البنية:
```
admins (المسؤولون)
    ↓
roles (الأدوار)
    ↓
role_permissions (ربط الأدوار بالصلاحيات)
    ↓
permissions (الصلاحيات)
```

#### كيف يعمل:
- ✅ **التحقق الثابت**: يتحقق من صلاحية المستخدم قبل تنفيذ العمل
- ✅ **المنع/السماح**: إما مسموح أو ممنوع
- ✅ **فوري**: القرار فوري (نعم/لا)

#### مثال:
```typescript
// التحقق: هل المستخدم لديه صلاحية حذف مخلفات؟
const canDelete = await checkWasteManagementPermission(userId, 'waste:catalog:delete');
if (!canDelete.allowed) {
  return; // منع العمل
}
// تنفيذ العمل مباشرة
```

#### الجداول المستخدمة:
- `permissions` - تعريف الصلاحيات
- `roles` - تعريف الأدوار
- `role_permissions` - ربط الأدوار بالصلاحيات
- `admins` - المسؤولون وربطهم بالأدوار

---

### 2. **نظام الموافقات (الجديد)**

#### البنية:
```
طلب عمل (مثل تغيير سعر)
    ↓
إنشاء طلب موافقة (pending)
    ↓
مدير المخلفات يراجع
    ↓
الموافقة/الرفض
    ↓
تطبيق العمل تلقائياً (إذا تمت الموافقة)
```

#### كيف يعمل:
- ✅ **Workflow متعدد المراحل**: pending → verified → approved/rejected
- ✅ **تسجيل وتتبع**: كل طلب يُسجل مع السبب والتاريخ
- ✅ **تطبيق تلقائي**: عند الموافقة، يتم التطبيق تلقائياً عبر Triggers
- ✅ **مراقبة**: يمكن تتبع جميع الطلبات وحالاتها

#### مثال:
```typescript
// المستخدم يطلب تغيير سعر (>= 10%)
// النظام يتحقق من الصلاحية أولاً
const canUpdate = await canManagePricing(userId, 'update_large');
if (!canUpdate.allowed) {
  return; // منع العمل
}

// إنشاء طلب موافقة (لا يتم التطبيق مباشرة)
const approval = await priceApprovalService.createApprovalRequest({
  waste_material_id: 'WASTE-001',
  old_price: 100,
  new_price: 120,
  price_change_percentage: 20,
  reason: 'زيادة الطلب',
  requested_by: userId,
  status: 'pending', // في انتظار الموافقة
});

// مدير المخلفات يوافق لاحقاً
await priceApprovalService.approveRequest(approval.id, managerId);
// ✅ يتم تطبيق تغيير السعر تلقائياً عبر Trigger
```

#### الجداول الجديدة:
- `waste_price_approval_requests` - طلبات الموافقة على الأسعار
- `waste_receiving_approval_requests` - طلبات الموافقة على الاستلام

#### الميزات الإضافية:
- **Triggers تلقائية**: تحديث الأسعار وتسجيل التاريخ عند الموافقة
- **RLS Policies**: أمان على مستوى الصفوف (كل مستخدم يرى طلباته فقط)
- **دوال مساعدة**: `requires_price_approval()` للتحقق من الحاجة للموافقة

## 📋 مقارنة سريعة

| الميزة | نظام الصلاحيات العام | نظام الموافقات (الجديد) |
|--------|---------------------|------------------------|
| **الهدف** | التحقق من الصلاحية | تسجيل وتتبع الموافقات |
| **التوقيت** | قبل العمل | قبل + بعد العمل |
| **القرار** | فوري (نعم/لا) | متعدد المراحل |
| **التسجيل** | لا | نعم (كل طلب يُسجل) |
| **التطبيق التلقائي** | لا | نعم (عبر Triggers) |
| **المراقبة** | محدودة | شاملة (جميع الطلبات) |
| **التاريخ** | لا | نعم (متى تمت الموافقة) |
| **السبب** | لا | نعم (سبب كل طلب) |

## 🎯 متى نستخدم كل نظام؟

### نظام الصلاحيات العام:
- ✅ **للعمليات البسيطة**: عرض، إضافة، تعديل، حذف
- ✅ **للعمليات الفورية**: لا تحتاج مراجعة
- ✅ **للتحكم الأساسي**: من يمكنه الوصول

### نظام الموافقات:
- ✅ **للعمليات الحساسة**: تغييرات كبيرة في الأسعار
- ✅ **للعمليات التي تحتاج مراجعة**: استلام المخلفات
- ✅ **للتتبع والمراقبة**: تتبع جميع التغييرات المهمة
- ✅ **للأمان الإضافي**: طبقة إضافية من التحقق

## 🔄 التكامل بين النظامين

### التدفق الكامل:

```
1. المستخدم يطلب عمل
   ↓
2. نظام الصلاحيات العام: هل لديه صلاحية؟
   ├─ لا → منع العمل
   └─ نعم → المتابعة
   ↓
3. نظام الموافقات: هل العمل يحتاج موافقة؟
   ├─ لا (< 10%) → تنفيذ مباشر
   └─ نعم (>= 10%) → إنشاء طلب موافقة
   ↓
4. مدير المخلفات يراجع
   ↓
5. الموافقة/الرفض
   ↓
6. Trigger يطبق التغيير تلقائياً (إذا تمت الموافقة)
```

## 💡 مثال عملي

### السيناريو: تغيير سعر مخلفات

#### بدون نظام الموافقات (النظام العام فقط):
```typescript
// التحقق من الصلاحية
if (hasPermission('waste:pricing:update')) {
  // تطبيق مباشر - لا توجد مراجعة!
  updatePrice(newPrice);
}
```

**المشاكل:**
- ❌ لا يوجد تتبع للتغييرات الكبيرة
- ❌ لا يوجد مراجعة قبل التطبيق
- ❌ لا يوجد تاريخ للتغييرات

#### مع نظام الموافقات:
```typescript
// 1. التحقق من الصلاحية (النظام العام)
if (!hasPermission('waste:pricing:update_large')) {
  return; // منع
}

// 2. حساب نسبة التغيير
const changePercent = calculateChange(oldPrice, newPrice);

// 3. إذا >= 10%، إنشاء طلب موافقة
if (changePercent >= 10) {
  createApprovalRequest({
    old_price: oldPrice,
    new_price: newPrice,
    reason: 'زيادة الطلب',
  });
  // لا يتم التطبيق مباشرة!
} else {
  // < 10%، تطبيق مباشر
  updatePrice(newPrice);
}

// 4. لاحقاً: مدير يوافق
approveRequest(requestId);
// ✅ Trigger يطبق التغيير تلقائياً
// ✅ يُسجل في exchange_price_history
```

**المزايا:**
- ✅ تتبع كامل للتغييرات
- ✅ مراجعة قبل التطبيق
- ✅ تاريخ كامل للتغييرات
- ✅ تطبيق تلقائي عند الموافقة

## 📊 الجداول في migration الجديد

### 1. `waste_price_approval_requests`
**الغرض**: تسجيل طلبات الموافقة على تغيير الأسعار

**الحقول المهمة:**
- `old_price`, `new_price` - السعر القديم والجديد
- `price_change_percentage` - نسبة التغيير
- `reason` - سبب التغيير
- `status` - الحالة (pending, approved, rejected)
- `requested_by`, `approved_by` - من طلب ومن وافق

### 2. `waste_receiving_approval_requests`
**الغرض**: تسجيل طلبات الموافقة على استلام المخلفات

**الحقول المهمة:**
- `delivery_agent_id` - الدليفري بوي الذي سلم
- `waste_items` - المخلفات المستلمة (JSONB)
- `status` - الحالة (pending_verification, verified, approved, rejected)
- `verified_by`, `approved_by` - من تحقق ومن وافق

## 🔐 RLS Policies (Row Level Security)

### ما الذي تضيفه؟

**بدون RLS:**
- أي مستخدم يمكنه رؤية جميع الطلبات (إذا كان لديه صلاحية عامة)

**مع RLS:**
- ✅ المستخدم يرى طلباته فقط
- ✅ مدير المخلفات يرى جميع الطلبات
- ✅ الدليفري بوي يرى طلباته فقط

**مثال:**
```sql
-- Policy: المستخدم يرى طلباته فقط
CREATE POLICY "Users can view their own price approval requests"
  ON public.waste_price_approval_requests
  FOR SELECT
  USING (auth.uid() = requested_by);
```

## ⚙️ Triggers التلقائية

### ما الذي تضيفه؟

**بدون Triggers:**
- يجب تحديث السعر يدوياً بعد الموافقة
- يجب تسجيل التاريخ يدوياً

**مع Triggers:**
- ✅ تحديث السعر تلقائياً عند الموافقة
- ✅ تسجيل التاريخ تلقائياً
- ✅ تحديث `stock_exchange` تلقائياً

**مثال:**
```sql
-- Trigger: عند الموافقة، يحدث السعر تلقائياً
CREATE TRIGGER trigger_update_stock_exchange_on_approval
  AFTER UPDATE ON public.waste_price_approval_requests
  FOR EACH ROW
  WHEN (NEW.status = 'approved' AND OLD.status = 'pending')
  EXECUTE FUNCTION public.update_stock_exchange_on_approval();
```

## 📝 الخلاصة

### نظام الصلاحيات العام:
- **التحقق**: هل يمكنك القيام بهذا؟
- **التطبيق**: فوري
- **التسجيل**: محدود

### نظام الموافقات (الجديد):
- **التحقق**: هل تمت الموافقة على هذا؟
- **التطبيق**: بعد الموافقة (تلقائياً)
- **التسجيل**: شامل (كل طلب مع السبب والتاريخ)

### التكامل:
- النظام العام يتحقق من الصلاحية
- نظام الموافقات يضيف طبقة إضافية من المراقبة والتتبع
- معاً يوفران أماناً وتتبعاً شاملاً
