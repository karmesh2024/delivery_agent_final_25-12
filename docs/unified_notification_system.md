# نظام الإشعارات الموحد (Unified Notification System)

هذا المستند يحدد آلية عمل نظام الإشعارات المركزي الذي يربط إدارات لوحة التحكم
(Dashboard) بتطبيقات المستخدمين والمناديب، باستخدام جدول `system_notifications`.

## 1. الهيكل المقترح للجدول (Schema Update)

لجعل الجدول يدعم تعدد الإدارات، نقترح تعديله ليتضمن الحقول التالية:

| الحقل               | النوع   | الوصف                                                               |
| :------------------ | :------ | :------------------------------------------------------------------ |
| `source_department` | `text`  | الإدارة المرسلة (مثل: `customer_mgmt`, `waste_mgmt`, `system_ops`). |
| `metadata`          | `jsonb` | بيانات إضافية (مثل: `order_id`, `promotion_code`, `action_link`).   |
| `priority`          | `text`  | أولوية الإشعار (`low`, `normal`, `high`).                           |

## 2. آلية العمل لكل إدارة وفئات المستخدمين

هذا النظام مصمم ليشمل جميع مستخدمي المنصة المربوطين بجدول `auth.users` العام.

### أ- إدارة العملاء (Customer Management)

- **المستهدفون**: العملاء في تطبيق "كارمش".
- **الحالة**: إرسال عروض، تنبيهات حساب، أو تحديثات السعر.
- **الآلية**: تحديد `target_user_id` للعميل المعني.

### ب- إدارة المناديب (Delivery/Courier Management)

- **المستهدفون**: مناديب التوصيل في تطبيق "مندوب كارمش".
- **الحالة**: تنبيه بطلب جديد، تحديث عمولة، أو رسالة تحذيرية.
- **الآلية**: استهداف المندوب الفردي أو الإرسال لجميع المناديب عبر
  `target_role = 'delivery'`.

### ج- إدارة الوكلاء (Agent Management)

- **المستهدفون**: الوكلاء وأصحاب مراكز الجمع.
- **الحالة**: تنبيهات تسلم شحنات، تسويات مالية، أو تغيير في سياسات القبول.
- **الآلية**: استهداف الوكيل المسؤول عن نقطة جغرافية محددة.

### د- إدارة المخلفات (Waste Management)

- **المستهدفون**: جميع الأطراف.
- **الحالة**: إخطار ببدء جلسة جمع، أو الانتهاء من الفوترة.
- **الآلية**: يتم الإرسال آلياً (Trigger) عند تغيير حالة الطلب.

### هـ- تحديثات النظام العامة

- **المستهدفون**: الجميع أو فئات مختارة.
- **الحالة**: صيانة، تحديثات تطبيقات، أو قوانين جديدة.
- **الآلية**: استخدام `target_role` (مثلاً: إرسال لكل من يحمل دور `agent`).

## 3. التكامل مع لوحة التحكم (Dashboard UI)

يجب توفير مكون واجهة مستخدم موحد (Reusable Component):

```typescript
// مكون مقترح لاستخدامه في أي صفحة داخل الداش بورد
<NotificationSender
  defaultTarget={userId}
  department="waste_mgmt"
  onSuccess={handleSuccess}
/>;
```

## 4. المسارات التقنية (API Routes)

سيتم استخدام المسار التالي كمركز لعمليات الإرسال:
`POST /api/notifications/system`

**مثال لطلب (Payload):**

```json
{
  "target_user_id": "customer-uuid-here",
  "title": "تم تحديث أسعار البلاستيك",
  "message": "عزيزي العميل، تم رفع نقاط تجميع البلاستيك اليوم!",
  "type": "price_update",
  "source_department": "waste_mgmt",
  "metadata": {
    "category": "plastic",
    "new_points": 50
  }
}
```

## 5. مراقبة الأداء (Logging)

يتم ربط كل إشعار يتم إرساله من الداش بورد بجدول `admin_activity_log` لمعرفة أي
موظف قام بإرسال الإشعار وفي أي وقت، لضمان المساءلة ومنع سوء الاستخدام.
