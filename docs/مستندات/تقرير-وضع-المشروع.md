## علاقة ClientOnly بـ API Supabase

بناءً على سؤال العميل، نوضح:

ClientOnly ليس له علاقة مباشرة بعدم الاتصال بـ API Supabase، بل يعمل كالتالي:

1. **وظيفة ClientOnly**: هو مكون يمنع تنفيذ المحتوى الداخلي له خلال عملية تصيير الخادم (Server Side Rendering - SSR)، ويضمن أن يتم تنفيذ المحتوى فقط في جانب العميل (المتصفح). هذا يحل مشكلة عدم التطابق (Hydration Mismatch) بين محتوى الخادم والعميل.

2. **الاتصال بـ Supabase**: يحدث الاتصال بـ API Supabase فعليًا، لكن فقط بعد تحميل الصفحة في المتصفح:
   - بدون ClientOnly: قد يحاول الكود الاتصال بـ Supabase في كل من الخادم والعميل، مما قد يسبب أخطاء.
   - مع ClientOnly: يتم تأجيل الاتصال بـ Supabase حتى يكتمل تحميل الصفحة في المتصفح.

3. **الحل الكامل**: إضافة توجيه "use client" + استخدام ClientOnly معًا يحل المشكلة، حيث:
   - "use client" يخبر Next.js أن هذا المكون خاص بجانب العميل فقط.
   - ClientOnly يضمن عدم تنفيذ المكون أثناء التصيير المبدئي للخادم.

## حل مشكلة Turbopack الحالية

الخطأ الحالي المتعلق بـ `[turbopack]/browser/dev/hmr-client/hmr-client.ts` هو مشكلة في أداة Turbopack نفسها وليس في الكود. لحل هذه المشكلة، يمكن اتباع أحد الخيارات التالية:

### 1. إعادة تشغيل خادم التطوير
```
npm run dev
```
أحيانًا إعادة تشغيل خادم التطوير يحل مشكلات Turbopack المؤقتة.

### 2. التبديل إلى Webpack بدلًا من Turbopack
تعديل ملف `next.config.mjs` لإيقاف استخدام Turbopack:
```javascript
/** @type {import('next').NextConfig} */
const nextConfig = {
  // إزالة أو تعطيل خيارات Turbopack
};

export default nextConfig;
```

### 3. مسح ذاكرة التخزين المؤقت
```
rm -rf .next
rm -rf node_modules
npm install
npm run dev
```

### 4. تحديث إصدار Next.js
قد تكون هذه المشكلة مصححة في إصدار أحدث من Next.js:
```
npm install next@latest
```
