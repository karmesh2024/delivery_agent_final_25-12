# تقرير هيكلة المشروع وخطة التحويل إلى DDD

## الوضع الحالي للمشروع

بعد تحليل بنية ملفات المشروع، وجدنا أنه في مرحلة انتقالية من الهيكل التقليدي لـ Next.js إلى هيكل التصميم الموجه بالنطاقات (DDD). وفيما يلي تحليل للوضع الحالي:

### المجلدات الرئيسية

1. **المجلدات التقليدية**:
   - `src/app`: يحتوي على صفحات Next.js (الطريقة التقليدية)
   - `src/components`: يحتوي على مكونات قديمة

2. **المجلدات المحدثة**:
   - `src/domains`: يحتوي على النطاقات المنظمة بشكل DDD
   - `src/shared`: يحتوي على المكونات والخدمات المشتركة

### حالة تحويل الصفحات

| الصفحة | مجلد app | مجلد domains | حالة التحويل |
|--------|----------|--------------|--------------|
| الرئيسية | `/page.tsx` | - | غير محول |
| الخريطة | `/map/page.tsx` | `/domains/mapping/pages/MapViewPage.tsx` | ✅ مكتمل |
| عرض الخريطة | `/map-view/page.tsx` | - | ❓ غير مكتمل |
| الوكلاء | `/agents/page.tsx` | `/domains/agents/pages/AgentsPage.tsx` | ✅ مكتمل |
| التحليلات | `/analytics/page.tsx` | `/domains/analytics/pages` | ❓ غير مكتمل |
| الطلبات | `/orders/page.tsx` | `/domains/orders/pages/OrdersPage.tsx` | ✅ مكتمل |
| تتبع الطلب | `/order-tracking/[id]/page.tsx` و `tracking-client.tsx` | - | ⚠️ تحويل جزئي |
| الرحلات | `/trips/page.tsx` و `trips-client.tsx` | `/domains/trips/pages` | ⚠️ تحويل جزئي |
| الرسائل | `/messages/page.tsx` | `/domains/messages` | ❓ غير مكتمل |
| الدعم | `/support/page.tsx` | `/domains/support/pages` | ❓ غير مكتمل |
| الإعدادات | `/settings/page.tsx` | `/domains/settings/pages` | ❓ غير مكتمل |

### المشاكل الحالية

1. **عدم اتساق في عملية التحويل**:
   - بعض الصفحات تم تحويلها بالكامل إلى هيكل DDD
   - بعض الصفحات تستخدم نهجًا مختلطًا
   - بعض الصفحات لم يتم تحويلها بعد

2. **ازدواجية في الملفات**:
   - وجود ملفات قديمة وجديدة تقدم نفس الوظائف
   - عدم وضوح أي الملفات يجب استخدامها

3. **عدم وضوح البنية النهائية**:
   - غموض في العلاقة بين ملفات `page.tsx` وملفات `*-client.tsx`
   - عدم اتساق في تسمية الملفات بين النطاقات المختلفة

## النموذج المثالي للتحويل

بناءً على نمط التحويل الناجح لصفحة الخريطة، يمكننا استخلاص النموذج المثالي للتحويل:

### مكونات النطاق (domains)

1. **هيكل المجلدات داخل النطاق**:
   ```
   src/domains/[domain]/
   ├── api/            # واجهات API الخاصة بالنطاق
   ├── components/     # مكونات خاصة بالنطاق
   ├── pages/          # صفحات النطاق
   ├── store/          # شرائح Redux
   ├── types/          # أنواع TypeScript
   └── utils/          # أدوات مساعدة
   ```

2. **ملف الصفحة في النطاق**:
   - يجب أن يكون مكون React كامل مستقل
   - يستورد المكونات من نفس النطاق أو من نطاقات أخرى
   - يتضمن كل المنطق الخاص بالصفحة
   - لا يستخدم `DashboardLayout` مباشرة

### ملفات Next.js (app)

1. **ملف الصفحة البسيط**:
   ```tsx
   // src/app/[route]/page.tsx
   import { DashboardLayout } from "@/shared/layouts/DashboardLayout";
   import DomainPage from "@/domains/[domain]/pages/DomainPage";

   export default function Page() {
     return (
       <DashboardLayout title="عنوان الصفحة">
         <DomainPage />
       </DashboardLayout>
     );
   }
   ```

2. **ملف الصفحة الديناميكي (مع معلمات)**:
   ```tsx
   // src/app/[route]/[id]/page.tsx
   import { DashboardLayout } from "@/shared/layouts/DashboardLayout";
   import DomainDetailPage from "@/domains/[domain]/pages/DomainDetailPage";

   export default function Page({ params }) {
     return (
       <DashboardLayout title="عنوان الصفحة">
         <DomainDetailPage id={params.id} />
       </DashboardLayout>
     );
   }
   ```

## خطة التحويل المنهجية

لإكمال تحويل المشروع إلى هيكل DDD بشكل متسق، نقترح الخطة التالية:

### 1. تحويل صفحات النطاقات المتبقية

#### صفحة تتبع الطلب (order-tracking)

1. **إنشاء ملف `OrderTrackingPage.tsx` في `domains/orders/pages`**:
   - نقل المنطق من `tracking-client.tsx`
   - استيراد `OrderTrackingView` من المكونات

2. **تبسيط ملف `app/order-tracking/[id]/page.tsx`**:
   - جعله يستورد `OrderTrackingPage` من النطاق

#### صفحة الرحلات (trips)

1. **إنشاء ملف `TripsPage.tsx` في `domains/trips/pages`**:
   - نقل المنطق من `trips-client.tsx`

2. **تبسيط ملف `app/trips/page.tsx`**:
   - جعله يستورد `TripsPage` من النطاق

#### باقي الصفحات (التحليلات، الرسائل، الدعم، الإعدادات)

1. **إنشاء ملفات الصفحات في نطاقاتها**:
   - نقل المنطق من ملفات `app`

2. **تبسيط ملفات `app`**:
   - جعلها تستورد الصفحات من النطاقات

### 2. توحيد تسمية الملفات

1. **توحيد تسمية الصفحات في النطاقات**:
   - استخدام نمط `[Domain]Page.tsx` (مثل `AgentsPage.tsx`, `OrdersPage.tsx`)
   - استخدام `[Domain]DetailPage.tsx` للصفحات التفصيلية

2. **التخلص من ملفات `*-client.tsx`**:
   - نقل المنطق إلى صفحات النطاقات
   - تبسيط ملفات `page.tsx` لتكون مجرد وسيط

### 3. حذف الملفات القديمة (بعد التأكد من نجاح التحويل)

1. **تحديد الملفات الزائدة**:
   - ملفات في مجلد `components` تم استبدالها بمكونات في النطاقات
   - ملفات `*-client.tsx` التي تم نقل منطقها
   - مكونات قديمة لم تعد مستخدمة

2. **إنشاء قائمة بالملفات للحذف**:
   - التأكد من عدم استخدامها أولاً
   - الحذف بعد التأكد من نجاح التحويل الكامل

## مثال تفصيلي لتحويل صفحة

لتوضيح عملية التحويل، هذا مثال تفصيلي لتحويل صفحة تتبع الطلب:

### 1. إنشاء `OrderTrackingPage.tsx` في `domains/orders/pages`:

```tsx
/**
 * صفحة تتبع الطلب ضمن نطاق الطلبات
 */

"use client";

import { useState, useEffect } from "react";
import { OrderTrackingView } from "../components/OrderTrackingView";
import { useAppDispatch } from "@/store/hooks";
import { fetchOrderById } from "../store/ordersSlice";

type OrderTrackingPageProps = {
  orderId: string;
};

const OrderTrackingPage = ({ orderId }: OrderTrackingPageProps) => {
  const dispatch = useAppDispatch();
  
  useEffect(() => {
    if (orderId) {
      dispatch(fetchOrderById(orderId));
    }
  }, [dispatch, orderId]);

  return <OrderTrackingView orderId={orderId} />;
};

export default OrderTrackingPage;
```

### 2. تبسيط `app/order-tracking/[id]/page.tsx`:

```tsx
import { Metadata } from "next";
import { DashboardLayout } from "@/shared/layouts/DashboardLayout";
import OrderTrackingPage from "@/domains/orders/pages/OrderTrackingPage";

type Props = {
  params: {
    id: string;
  };
};

export const metadata: Metadata = {
  title: "تتبع الطلب | لوحة تحكم منصة التوصيل",
  description: "صفحة تتبع الطلب والمسارات بين المندوب والعميل",
};

export default async function Page({ params }: Props) {
  const orderId = params?.id || '';
  
  return (
    <DashboardLayout title="تتبع الطلب">
      <OrderTrackingPage orderId={orderId} />
    </DashboardLayout>
  );
}
```

## ملخص عن هيكل المشروع النهائي

بعد اكتمال تحويل جميع الصفحات، سيكون هيكل المشروع النهائي كالتالي:

1. **مجلد `src/app`**:
   - يحتوي فقط على ملفات Next.js البسيطة
   - كل ملف `page.tsx` يستورد صفحة من نطاق معين ويغلفها بـ `DashboardLayout`
   - لا يحتوي على منطق معقد

2. **مجلد `src/domains`**:
   - كل نطاق يحتوي على مكوناته ومنطقه الخاص
   - الصفحات في مجلد `pages` داخل كل نطاق
   - البيانات تُدار بواسطة Redux في مجلد `store` داخل كل نطاق

3. **مجلد `src/shared`**:
   - يحتوي على المكونات والخدمات المشتركة بين النطاقات
   - يتضمن تخطيط التطبيق الرئيسي `DashboardLayout`

4. **مجلد `src/store`**:
   - يحتوي على تكوين متجر Redux المركزي
   - يجمع شرائح النطاقات المختلفة

## الخطوات التالية

1. **تنفيذ خطة التحويل بالترتيب التالي**:
   - صفحات تتبع الطلب والرحلات التي بدأت في التحويل
   - الصفحات المتبقية (التحليلات، الرسائل، الدعم، الإعدادات)
   - توحيد التسميات والأنماط
   - حذف الملفات الزائدة

2. **توثيق الهيكل النهائي**:
   - إنشاء دليل للمطورين حول كيفية إضافة صفحات جديدة
   - توضيح العلاقة بين النطاقات المختلفة

3. **إنشاء اختبارات**:
   - التأكد من أن التحويل لم يؤثر على وظائف التطبيق

بهذه الطريقة، سيكون لدينا مشروع منظم بشكل جيد يتبع مبادئ التصميم الموجه بالنطاقات (DDD) ويسهل صيانته وتطويره في المستقبل.
