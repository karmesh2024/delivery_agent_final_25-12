# الدليل الشامل لتحويل المشروع للهيكل الجديد

هذا الدليل يوضح الخطوات المنهجية لاستكمال تحويل مشروع DeliverEase من الهيكل القديم إلى الهيكل الجديد المبني على نهج تصميم النطاقات (Domain-Driven Design). الهدف هو ضمان العمل فقط بالمكونات المحولة وإزالة أو تعليق الملفات القديمة غير الضرورية.

## الوضع الحالي للمشروع

حالياً، المشروع في مرحلة انتقالية حيث:

1. **الهيكل الجديد**: تم إنشاء هيكل مجلدات جديد (`src/domains/`, `src/shared/`) ونقل بعض المكونات إليه.
2. **واجهات API**: تم إنشاء واجهات API مجردة (مثل `agentsApi`, `ordersApi`) كطبقة وسيطة للوصول لقاعدة البيانات.
3. **Redux**: تم إنشاء شرائح Redux لإدارة الحالة المركزية.
4. **الصفحات**: لا تزال صفحات التطبيق في `src/app` تستخدم مزيجاً من:
   - مكونات من الهيكل الجديد
   - استدعاءات مباشرة لـ `lib/supabase.ts` بدلاً من واجهات API الجديدة
   - حالة محلية (`useState`) بدلاً من Redux

## خطة التحويل المنهجية

### المرحلة 1: تحديث صفحة الداشبورد الرئيسية

1. **تحديث استيرادات API**:

2. **تطبيق Redux واستخدام hooks**:
   ```typescript
   // قبل
   const [agents, setAgents] = useState<Agent[]>([]);
   const [orders, setOrders] = useState<Order[]>([]);
   
   // بعد
   import { useAppDispatch, useAppSelector } from "@/store/hooks";

   import { fetchOrders } from "@/domains/orders/store/ordersSlice";

   const { items: orders, loading: ordersLoading } = useAppSelector(state => state.orders);
   ```

3. **تحديث استدعاءات البيانات**:
   ```typescript

       // ...

   }
   
   // بعد

     dispatch(fetchAgents());

   }, [dispatch]);
   ```

```typescript
  function calculateAverageDeliveryTime(orders: Order[]) {
    const deliveredOrders = orders.filter(o => o.status === 'delivered' && o.delivery_time);
    if (deliveredOrders.length === 0) return 0;
    
    const totalTime = deliveredOrders.reduce((sum, order) => sum + (order.delivery_time || 0), 0);
    return Math.round(totalTime / deliveredOrders.length);
```

## أفضل الممارسات لضمان نجاح التحويل

1. **استخدام التسميات الموحدة**: تأكد من استخدام نفس التسميات والأنماط في جميع الملفات الجديدة.

2. **الحفاظ على الواجهات العامة**: تأكد من أن واجهات المكونات الجديدة متوافقة مع المكونات القديمة لتسهيل التحويل.

3. **الاتساق في نطاق الاستعمال**: عندما تحول ملفاً، تأكد من استخدام المكونات والواجهات الجديدة باستمرار.

4. **التوثيق**: أضف تعليقات واضحة في كل ملف جديد لشرح الغرض منه وكيفية استخدامه.

5. **التحكم بالنسخ**: استخدم Git لتتبع التغييرات وإمكانية التراجع عنها إذا لزم الأمر.

## استراتيجية إلغاء الملفات القديمة

بعد استكمال تحويل الصفحات للنموذج الجديد، اتبع هذه الخطوات لإلغاء الملفات القديمة تدريجياً:

1. **تحديد الملفات المستهدفة**: استخدم القائمة المذكورة أعلاه للملفات التي يجب تعليقها.

2. **إنشاء ملفات توجيه**: لكل ملف قديم، قم بتحويله إلى ملف توجيه يشير إلى البديل الجديد.

3. **اختبار شامل**: بعد كل تغيير، قم باختبار جميع وظائف التطبيق للتأكد من عدم وجود مشكلات.

4. **الإزالة النهائية**: بعد التأكد من عمل كل شيء بشكل صحيح، يمكن إزالة الملفات القديمة تماماً.

## خاتمة

تعتبر عملية تحويل المشروع للهيكل الجديد خطوة مهمة لتحسين قابلية التوسع والصيانة. باتباع هذا الدليل المنهجي، يمكن إكمال التحويل بطريقة منظمة وآمنة، مع ضمان عمل النظام بالمكونات المحولة فقط وإزالة الملفات القديمة غير الضرورية.
