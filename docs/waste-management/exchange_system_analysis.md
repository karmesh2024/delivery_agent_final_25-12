# تحليل نظام تسعير البورصة والمشاكل الحالية

هذا المستند يوضح الحالة الحالية لنظام تسعير البورصة في تطبيق Delivery Agent
Dashboard، مع التركيز على مشكلة عدم ظهور مؤشرات التغيير للفئات الفرعية، وتقييم
شامل لاكتمال النظام.

## 1. المشكلة: غياب مؤشر التغيير للفئات الفرعية

لاحظنا أن **شريط أسعار البورصة (Ticker)** يعرض الفئات الفرعية (مثل "زجاجات"،
"بلاستيك"، إلخ) ولكن مؤشر التغيير الخاص بها يظهر دائماً كـ `0.00` و `(0.0%)`،
بينما يظهر للمنتجات الفردية بشكل صحيح.

### السبب التقني الجذري

بعد فحص الكود المصدري، تبين الآتي:

1. **غياب جدول لتاريخ أسعار الفئات**: النظام الحالي يحتوي على جدول
   `exchange_price_history` لتسجيل تغيرات أسعار **المنتجات الفردية** فقط. لا يوجد
   جدول مماثل (مثل `subcategory_price_history`) لتسجيل تغيرات أسعار **الفئات
   الفرعية** عبر الزمن.

2. **الكود المثبت (Hardcoded)**: في ملف `PriceTicker.tsx`، يتم تعيين قيم التغيير
   للفئات الفرعية يدوياً إلى صفر لأن النظام لا يوفر بيانات للمقارنة:
   ```typescript
   // في PriceTicker.tsx
   change: 0,          // ثابت
   changePercent: 0,   // ثابت
   type: 'neutral'     // ثابت
   ```

3. **آلية التحديث**: دالة تحديث السعر `setSubcategoryExchangePrice` تقوم فقط
   بتحديث السعر الحالي في جدول `subcategory_exchange_price` ولا تحتفظ بالسعر
   القديم في أي سجل تاريخي، مما يجعل من المستحيل حساب نسبة التغيير (السعر
   الجديد - السعر القديم).

## 2. تقييم النظام الحالي للبورصة

### نقاط القوة ✅

- **تسعير المنتجات**: يعمل بشكل جيد. يتم تسجيل تاريخ التغييرات، وتظهر المؤشرات
  (صعود/هبوط) في الشريط.
- **الربط الهرمي**: تم إصلاح مشكلة ربط المنتجات بفئاتها، والآن تظهر المنتجات تحت
  فئاتها الصحيحة.
- **التحديث الفوري**: النظام يدعم التحديثات المباشرة (Real-time).

### نقاط الضعف والنواقص ❌

- **تاريخ أسعار الفئات (مفقود)**: لا يمكن تتبع كيف تغير سعر "النحاس" أو "الزجاج"
  كفئة عامة خلال اليوم أو الأسبوع.
- **عدم اكتمال الدورة المستندية**: عند تغيير سعر فئة، يتم تغيير أسعار المنتجات
  التابعة لها، ولكن لا يتم تسجيل "حدث التغيير" للفئة نفسها كمرجع.
- **الاعتماد على أسعار المنتجات فقط**: المؤشرات العامة للسوق يجب أن تُبنى على
  "مؤشرات الفئات" (Category Indices) وليس فقط المنتجات الفردية.

## 3. هل النظام كامل وصحيح؟

**الإجابة: النظام "وظيفي" (Functional) ولكنه "غير كامل" (Incomplete) من ناحية
تحليل البيانات والتوثيق التاريخي للفئات.**

هو "صحيح" بمعنى أنه لا يعطي أخطاء برمجية قاتلة (Crashes)، ولكنه "غير كامل" لأنه
يفتقر لميزة أساسية في أي نظام بورصة، وهي **تتبع تاريخ المؤشرات الرئيسية
(الفئات)**.

## 4. خطة الإصلاح المقترحة (Roadmap)

لجعل النظام مكتملاً، نقترح الخطوات التالية:

### أ. إنشاء جدول لتاريخ أسعار الفئات

يجب إنشاء جدول جديد في قاعدة البيانات:

```sql
CREATE TABLE subcategory_price_history (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  subcategory_id BIGINT REFERENCES waste_sub_categories(id),
  old_buy_price NUMERIC,
  new_buy_price NUMERIC,
  old_sell_price NUMERIC,
  new_sell_price NUMERIC,
  changed_by UUID REFERENCES auth.users(id),
  change_reason TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

### ب. تعديل خدمة التسعير

تعديل دالة `setSubcategoryExchangePrice` في `subcategoryExchangePriceService.ts`
لتقوم بإضافة سجل في الجدول الجديد عند كل تغيير للسعر.

### ج. تحديث الواجهة (Price Ticker)

تعديل `PriceTicker.tsx` لجلب بيانات التغيير من الجدول الجديد وحساب النسبة
المئوية للفئات، تماماً كما يحدث مع المنتجات.

---

**الخلاصة:** الوضع الحالي طبيعي في مرحلة التطوير، ولكن لإطلاق نظام بورصة "احترافي"،
فإن تفعيل تاريخ أسعار الفئات هو خطوة ضرورية.
