# سياسة تحويل نقاط المخلفات إلى نقاط النادي - إصدار V1.3 المبسّط

## 🎯 الهدف

تحديد **قواعد بسيطة وآمنة** لتحويل جزء من نقاط المخلفات إلى نقاط النادي في المرحلة الأولى (V1.3)، بدون تعقيد، وبدون فتح باب لاستنزاف منظومة الجوائز.

---

## 1️⃣ ما المقصود بـ Admin Only + نسبة ثابتة + سقف شهري + لا تحويل ذاتي؟

### 1. Admin Only (التحويل عن طريق الأدمن فقط)

المعنى:
- المستخدم (العضو) **لا يملك زر** في التطبيق لتحويل نقاط المخلفات إلى نقاط نادي.
- التحويل يتم فقط من خلال:
  - **لوحة تحكم داخلية** يراها الأدمن / موظف الجمعية.
  - أو **سكريبت/أداة داخلية** يشغّلها الأدمن شهريًا.

الهدف:
- تجنّب:
  - ضغط مفاجئ من المستخدمين لتحويل كل المخلفات إلى نقاط نادي.
  - قرارات غير محسوبة تؤثر على تكلفة الجوائز.
- إعطاء الجمعية تحكم كامل في:
  - من يتم تحويل نقاطه.
  - نسبة التحويل الفعلية.
  - توقيت التحويل (مرة شهريًا مثلاً).

#### ✅ تنظيم العملية بدون “تحويل ذاتي”: نموذج طلب تحويل (Request Only)

في V1.3 ممكن نضيف **تنظيم بسيط** يقلل الضغط على الأدمن بدون ما نفتح تحويل تلقائي/ذاتي:

- المستخدم يعمل **طلب تحويل فقط** (لا تنفيذ).
- الأدمن يراجع الطلبات ويوافق/يرفض.
- عند الموافقة يتم التنفيذ يدويًا (خصم من نظام المخلفات + إضافة نقاط نادي).

```sql
CREATE TABLE IF NOT EXISTS recycling_conversion_requests (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES new_profiles(id),
  recycling_points INTEGER NOT NULL,
  club_points_expected INTEGER NOT NULL, -- حساب ثابت (مثلاً 30%)
  status VARCHAR(20) NOT NULL DEFAULT 'PENDING' CHECK (status IN (
    'PENDING',
    'APPROVED',
    'REJECTED'
  )),
  processed_by UUID REFERENCES new_profiles(id),
  processed_at TIMESTAMPTZ,
  rejection_reason TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

> هذا لا يُعتبر “تحويل ذاتي” لأنه لا يتم إلا بعد موافقة الأدمن، وهو مجرد سجل/Queue لتنظيم العمل.

---

### 2. نسبة ثابتة (مثلاً 30%)

المعنى:
- يتم الاتفاق داخليًا أن **كل 100 نقطة مخلفات** تعادل **30 نقطة نادي** مثلاً.
- النسبة ثابتة في الكود أو في إعداد واحد (وليس لكل مستخدم أو لكل حالة).

مثال:
- عضو لديه 250 نقطة مخلفات.
- إذا قرر الأدمن تحويل جزء منها:
  - لو حوّل 100 نقطة مخلفات → يحصل العضو على 30 نقطة نادي.
  - لو حوّل 200 نقطة مخلفات → يحصل على 60 نقطة نادي.

أين تُستخدم هذه النسبة؟
- في منطق backend فقط، مثلاً:

```pseudo
club_points = floor(waste_points * 0.30)
```

> لا يوجد في V1.3 نسب مختلفة لكل شريحة أو لكل حملة؛ نسبة واحدة بسيطة للجميع.

---

### 3. سقف شهري بسيط (Hardcoded أو ثابت في الكود)

المعنى:
- حتى لو كان للمستخدم نقاط مخلفات كثيرة، **لن نسمح** بتحويل كمية غير محدودة في شهر واحد.
- نضع حدًا ثابتًا واضحًا، مثلاً:
  - أقصى ما يمكن تحويله إلى نقاط نادي = **2000 نقطة نادي لكل مستخدم في الشهر**.

لماذا؟
- حتى لو حدث خطأ إداري أو تسرّع في قرار، لا يمكن أن:
  - يفرّغ مستخدم واحد مخزون الجوائز.
  - تتضاعف تكلفة الجوائز فجأة في شهر واحد.

كيف يُطبَّق في V1.3؟
- في الكود (أو Function بسيطة) نحسب:
  - **كم نقطة نادي تم منحها لهذا المستخدم من نوع `waste_conversion` خلال هذا الشهر**.
  - لو مجموع (الحالي + الجديدة) > السقف الشهري:
    - نرفض العملية، أو نقلّل الكمية بحيث لا تتجاوز السقف.

Pseudo مثال:

```pseudo
monthly_converted = SUM(points)
  FROM club_points_transactions
  WHERE user_id = X
    AND source = 'waste_conversion'
    AND month(created_at) = current_month

if (monthly_converted + new_club_points) > MONTHLY_CAP:
  reject or adjust
else
  allow
```

> في V1.3 يكفي أن يكون هذا المنطق **مكتوبًا داخل Service** أو Function واحدة، وليس منظومة سياسات معقّدة.

---

### 4. لا يوجد تحويل ذاتي/تلقائي للمستخدم

#### أ. لا تحويل ذاتي (No Self Conversion)

المعنى:
- لا يوجد في واجهة المستخدم:
  - زر "حوّل نقاط مخلفاتك إلى نقاط نادي".
  - ولا شاشة تتيح للمستخدم اختيار الكمية والتحويل بنفسه.

كل التحويلات:
- تأتي من جهة الأدمن فقط (من لوحة تحكم/سكربت داخلي).

#### ب. لا تحويل تلقائي (No Auto Conversion)

المعنى:
- النظام نفسه **لن ينفّذ تحويلًا تلقائيًا** في الخلفية كلما زادت نقاط المخلفات.
- لا يوجد:
  - Job شهري يحوّل تلقائيًا نسبة من نقاط المخلفات لكل المستخدمين.
  - Trigger في قاعدة البيانات يحوّل كل عملية مبيع مخلفات إلى نقاط نادي.

لماذا الاثنين معًا (لا ذاتي + لا تلقائي) مهمّان في V1.3؟
- لأنّ:
  - **كل نقطة نادي = تكلفة على منظومة الجوائز** (هدايا/كروت شحن/منتجات).
  - في مرحلة الإطلاق نريد:
    - رؤية الأرقام يدويًا.
    - تجربة النموذج المالي على عيّنات صغيرة.
    - ضبط النسبة (30% مثلاً) قبل تعميمها أو أتمتتها في V2.5.

## 2️⃣ سيناريو عملي كامل لتحويل المخلفات في V1.3

### 🎬 السيناريو: عضو يريد تحويل نقاط مخلفات إلى نقاط نادي

**البيانات:**
- العضو: أحمد.
- نقاط المخلفات المتاحة في نظام المخلفات: 800 نقطة.
- نسبة التحويل المتفق عليها: 30%.
- السقف الشهري لتحويل نقاط النادي: 2000 نقطة نادي لكل مستخدم.
- هذا هو التحويل الأول لأحمد هذا الشهر (لا توجد تحويلات سابقة).

### الخطوات (من وجهة نظر الأدمن):

1. **الأدمن يفتح لوحة تحكم داخلية (خاصة بالجمعية)**:
   - يدخل على صفحة "تحويل نقاط المخلفات".

2. **البحث عن العضو أحمد**:
   - عبر رقم الهاتف / ID / الاسم.
   - النظام يعرض:
     - نقاط المخلفات المتاحة (800).
     - نقاط النادي الحالية:
       - Pending / Available / Used (من `club_points_wallet`).
     - كمية التحويل السابقة هذا الشهر من نوع `waste_conversion` (من `club_points_transactions`).

3. **اختيار كمية نقاط المخلفات المراد تحويلها**:
   - الأدمن يقرر مثلاً تحويل 300 نقطة مخلفات فقط.

4. **النظام يحسب الكمية المتاحة ضمن السقف الشهري**:

   - أولاً: حساب ما تم تحويله هذا الشهر:
     - لنفترض أنه 0 (أول مرة).
   - ثانيًا: حساب نقاط النادي الناتجة عن التحويل:

   \[
   \text{club\_points} = 300 \times 0.30 = 90
   \]

   - ثالثًا: التحقق من السقف الشهري:
     - `monthly_converted_so_far = 0`
     - `new_club_points = 90`
     - `monthly_cap = 2000`
     - \( 0 + 90 \le 2000 \) → العملية مسموحة.

5. **تثبيت العملية**:
   - (جزء المخلفات):
     - يتم خصم 300 نقطة من نظام المخلفات (منظومة أخرى منفصلة).
   - (جزء النادي):
     - استدعاء:
       - `update_club_points_wallet(user_id=Ahmed, points=90, transaction_type='EARNED', source='waste_conversion', description='تحويل 300 نقطة مخلفات إلى 90 نقطة نادي')`.
     - النتيجة في `club_points_wallet`:
       - `pending_points` تزيد بمقدار 90.
       - `lifetime_points` تزيد بمقدار 90.
     - إنشاء سجل في `club_points_transactions`:
       - `transaction_type = 'EARNED'`
       - `points = 90`
       - `source = 'waste_conversion'`
       - `points_before / points_after` على أساس `available_points` (حسب التصميم).

6. **العضو يرى النتيجة في تطبيق النادي**:
   - في شاشة النقاط:
     - `نقاط معلّقة (PENDING):` زادت بـ 90.
     - ينتظر الاعتماد الشهري ليتم تحويلها إلى `AVAILABLE`.

7. **في نهاية الشهر**:
   - الأدمن يضغط زر: "اعتماد نقاط الشهر".
   - يتم استدعاء: `activate_monthly_points('2026-01', admin_id, 'اعتماد نقاط الشهر')`.
   - جميع النقاط المعلّقة (بما فيها الـ 90 نقطة من المخلفات) تتحوّل إلى:
     - `available_points`.
   - العضو الآن يمكنه استخدام هذه النقاط في استبدال الجوائز.

## 3️⃣ لماذا هذا الأسلوب آمن في V1.3؟

1. **لا يوجد تحويل مفاجئ وكبير**:
   - لأن الأدمن هو من يختار من يتم تحويل نقاطه ومتى.

2. **النسبة واضحة وبسيطة**:
   - 30% ثابتة، لا تعقيد في الحساب.

3. **السقف الشهري يمنع الانفجار**:
   - حتى لو تم تحويل كمية كبيرة من المخلفات إداريًا، لن تتجاوز مقابلها في نقاط النادي سقفًا معينًا لكل عضو في الشهر.

4. **مرحلة اختبار مالي**:
   - يمكن للجمعية بعد 3–6 أشهر أن تراجع:
     - كم نقطة نادي خرجت من التحويل.
     - كم جائزة خرجت بسبب هذه النقاط.
   - ثم تقرر:
     - هل نسبة 30% مناسبة؟
     - هل السقف الشهري 2000 مناسب أم يحتاج تعديل؟
     - هل يمكن فتح جزء من التحويل للمستخدمين ذاتيًا في V2.5؟

---

## 4️⃣ ملخص السياسة في جملة واحدة

**في V1.3**:  
تحويل نقاط المخلفات إلى نقاط النادي يتم **يدويًا فقط بواسطة الأدمن**، **بنسبة ثابتة بسيطة**، **مع سقف شهري واضح لكل عضو**، و**بدون أي تحويل ذاتي أو تلقائي للمستخدمين**، لضمان حماية منظومة الجوائز في مرحلة الإطلاق.
