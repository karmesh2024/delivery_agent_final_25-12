# رأي في السيناريو المقترح (السيناريو الكامل المُحدَّث)

هذا المستند يوضح **رأيي في المقترح** ومقارنته بما **مُنفَّذ حالياً** في المشروع، وما يُفضَّل **تبنيه أو تكييفه**.

---

## 1. الرأي العام: المقترح ممتاز ومحقق للأهداف

المقترح **منطقي ومتكامل** ويحقق الأهداف الثمانية التي ذكرتها:

| الهدف | المقترح | الرأي |
|-------|---------|--------|
| إدارة التسلسل = المصدر الأساسي | ✅ | **موافق** |
| المزامنة التلقائية مع الكتالوج | ✅ Trigger أو Service | **موافق** |
| إدارة المخازن تعتمد على الكتالوج | ✅ catalog_waste_id | **موافق** |
| البورصة تعرض الأسعار | ✅ stock_exchange + catalog | **موافق** |
| تسعير مرن (فئة vs منتج) | ✅ subcategory_exchange_price + price_per_kg | **موافق** |
| تعديل أسعار البورصة + موافقة ≥10% | ✅ | **موافق** — **منفَّذ فعلياً** |
| التطبيقات تعرض البيانات الصحيحة | ✅ | **موافق** |
| حذف آمن (Soft Delete) | ✅ | **موافق** — **منفَّذ فعلياً** |

**الخلاصة:** السيناريو المُقترح **أساس سليم** ويمكن اعتماده مع تكييف أسماء الحقول والجداول ليتوافق مع الـ Schema الحالي.

---

## 2. مقارنة المقترح مع التنفيذ الحالي

### 2.1 أسماء الحقول في المشروع الحالي

المقترح يستخدم أسماء قد تختلف قليلاً عن الـ Schema الحالي:

| المقترح | المشروع الحالي | ملاحظة |
|---------|-----------------|--------|
| `waste_main_category_id` في waste_data_admin | **لا يوجد** — يوجد `category_id` (UUID) و `subcategory_id` (BigInt) | الفئة الأساسية تُستنتج من الفئة الفرعية (waste_sub_categories.main_id) |
| `waste_sub_category_id` في waste_data_admin | `subcategory_id` (BigInt) | نفس المفهوم، اسم مختلف |
| `catalog_waste_id` في waste_data_admin | **لا يوجد في waste_data_admin** | موجود في `warehouse_inventory` و `purchase_order_items` — **يُفضَّل إضافته لـ waste_data_admin** |
| `main_category_id` في waste_sub_categories | `main_id` (BigInt) | نفس المفهوم |
| `price_per_kg` في subcategory_exchange_price | `buy_price` و `sell_price` | المقترح يستخدم price_per_kg؛ عندنا buy/sell — **لا تعارض** |
| `current_price` / `previous_price` في stock_exchange | `buy_price` و `base_price` | نفس الفكرة، أسماء مختلفة |
| `catalog_waste_id` في stock_exchange | **لا يوجد** | **اختياري** — الربط الحالي عبر `product_id` (waste_data_admin) كافٍ |

### 2.2 ما مُنفَّذ حالياً في الكود

| العنصر | الحالة | الملف / الملاحظة |
|--------|--------|-------------------|
| مزامنة منتج جديد → كتالوج | ✅ | `wasteCatalogSyncService.syncNewProduct` + استدعاء من `productService.createProduct` |
| مزامنة تحديث منتج → كتالوج | ✅ | `wasteCatalogSyncService.syncUpdatedProduct` من `productService.updateProduct` |
| مزامنة حذف (تعطيل) → كتالوج | ✅ | `wasteCatalogSyncService.syncDeletedProduct` (status: 'deleted') من `productService.deleteProduct` |
| مزامنة سعر الفئة → أسعار البورصة | ✅ | `subcategoryExchangePriceService` عند تحديث سعر الفئة الفرعية |
| منطق الموافقة ≥10% للفئة الفرعية | ✅ | `updateSubcategoryPriceWithApprovalCheck` + `subcategory_price_approval_requests` |
| تسجيل تاريخ الأسعار | ✅ | `exchange_price_history` من `exchangeService.logPriceHistory` |
| إنشاء فئة فرعية مع سعر أولي | ⏳ | غير مضمّن في نقطة واحدة — يمكن دمجه عند إنشاء الفئة |

### 2.3 ما يختلف أو ما يُقترح إضافته

| المقترح | الوضع الحالي | التوصية |
|---------|--------------|----------|
| **Trigger في DB** للمزامنة عند UPDATE على waste_data_admin | مزامنة من **التطبيق** (productService) فقط | **الخياران مقبولان.** الـ Trigger يضمن المزامنة حتى لو تم التحديث من مكان آخر؛ التطبيق أخف ولا يحتاج migrations إضافية. يمكن لاحقاً إضافة Trigger إذا رغبت. |
| **ربط عكسي:** waste_data_admin.catalog_waste_id | العمود **غير موجود** في waste_data_admin حالياً | **مُوصى به:** إضافة عمود `catalog_waste_id` في waste_data_admin وتحديثه عند إنشاء سجل الكتالوج؛ يسهل الاستعلام والربط مع المخازن. |
| **إنشاء فئة فرعية مع سعر أولي** في خطوة واحدة | إنشاء الفئة ثم تعيين السعر من واجهة أسعار الفئات الفرعية | **مُوصى به:** دالة واحدة `createSubCategoryWithPrice` (أو ما يعادلها في الواجهة) لإنشاء الفئة + سجل في subcategory_exchange_price. |
| **إنشاء منتج مع ربط كتالوج + بورصة** في transaction واحدة | إنشاء المنتج ثم المزامنة إلى الكتالوج (بدون ربط عكسي)؛ البورصة تُدار من مسار آخر | **مُوصى به:** عند إنشاء المنتج: إنشاء الكتالوج، تعيين waste_data_admin.catalog_waste_id، وإنشاء/ربط stock_exchange إن لزم — مع الحفاظ على التوافق مع stock_exchange (category_id/subcategory_id UUID إن وُجدت). |
| **Soft Delete** يعطّل أيضاً stock_exchange و catalog | عندنا تعطيل الكتالوج فقط (status)؛ حذف المنتج من waste_data_admin حذف فعلي | **مُوصى به:** توحيد السلوك: إما soft delete للمنتج (تعطيل ظهور فقط) مع تعطيل سجل البورصة والكتالوج، أو الإبقاء على الحذف الفعلي مع تعطيل الكتالوج فقط (كما هو الآن). |
| **سكريبت المزامنة الأولية** للمنتجات بدون catalog_waste_id | يوجد `syncAllProducts()` في wasteCatalogSyncService لكنه لا يملأ catalog_waste_id (العمود غير موجود) | **مُوصى به:** بعد إضافة catalog_waste_id: توسيع المزامنة الأولية لتعيين catalog_waste_id في waste_data_admin بعد إنشاء/ربط سجل الكتالوج. |

## 3. التوصيات العملية

### 3.1 يمكن اعتماد المقترح كما هو من ناحية **السيناريو والمنطق**

- إنشاء فئة فرعية مع سعر أولي.
- إنشاء منتج مع كتالوج + ربط + بورصة.
- تحديث أسعار البورصة مع موافقة ≥10%.
- مزامنة التحديثات مع الكتالوج (Trigger أو Service).
- Soft Delete بدون فقدان بيانات.

كل ذلك **متوافق مع أهدافك** وما بدأنا ننفذه.

### 3.2 تكييفات ضرورية لـ Schema الحالي

1. **أسماء الحقول:**  
   استخدام `subcategory_id` و `main_id` (من الفئة الفرعية) بدلاً من `waste_sub_category_id` و `waste_main_category_id` في waste_data_admin.

2. **إضافة `catalog_waste_id` إلى waste_data_admin (اختياري لكن مفيد):**
   - Migration: `ALTER TABLE waste_data_admin ADD COLUMN catalog_waste_id BIGINT REFERENCES catalog_waste_materials(id);`
   - عند إنشاء سجل الكتالوج من المزامنة، تحديث المنتج: `UPDATE waste_data_admin SET catalog_waste_id = ? WHERE id = ?`.

3. **جدول stock_exchange الحالي:**  
   يستخدم `category_id` و `subcategory_id` (UUID). إضافة `catalog_waste_id` إلى stock_exchange **اختيارية** إذا أردت ربطاً مباشراً بالكتالوج؛ وإلا يبقى الربط عبر `product_id` كافياً.

4. **Trigger المزامنة:**  
   إذا أضفت Trigger، يجب أن يستخدم الأعمدة الحقيقية: `subcategory_id`, `name`, `price_per_kg`, `weight`، وأن يحدّث `catalog_waste_materials` حيث `id = waste_data_admin.catalog_waste_id` (بعد إضافة العمود).

### 3.3 ما يمكن تأجيله أو تركه كتحسين لاحق

- إضافة `catalog_waste_id` إلى `stock_exchange` إن كان الربط الحالي عبر `product_id` كافياً لك.
- Trigger المزامنة إن كنت واثقاً أن كل التحديثات تمر عبر `productService.updateProduct`.

## 4. خطة تنفيذ مقترحة (مختصرة) — وحالة التنفيذ الفعلية

| الخطوة | الإجراء | الأولوية | **الحالة** |
|--------|---------|----------|-------------|
| 1 | توثيق السيناريو الكامل في مستند واحد (مثل `سيناريو_المزامنة_قبل_وبعد.md`) مع تضمين إنشاء فئة فرعية مع سعر، وإنشاء منتج مع كتالوج وبورصة | عالية | ✅ **منفَّذ** — الملف موجود |
| 2 | إضافة عمود `catalog_waste_id` إلى waste_data_admin + تحديث wasteCatalogSyncService لملئه عند الإنشاء/المزامنة | عالية | ✅ **منفَّذ** — migration `20260201_add_catalog_waste_id_to_waste_data_admin.sql` + الخدمة تملأ الحقل عند الإنشاء والتحديث |
| 3 | توسيع إنشاء المنتج: بعد المزامنة مع الكتالوج، استدعاء تحديث لتعيين catalog_waste_id، والربط مع stock_exchange حسب السياسة الحالية | عالية | ✅ **منفَّذ** — تعيين catalog_waste_id يتم داخل `syncNewProduct` عبر `setProductCatalogId`. ربط stock_exchange يبقى عبر المسار الحالي (مثلاً createProductWithOpeningPrice) |
| 4 | (اختياري) دالة أو واجهة لـ «إنشاء فئة فرعية مع سعر أولي» | متوسطة | ✅ **منفَّذ** — `categoryService.createSubCategoryWithInitialExchangePrice` + واجهة في نموذج الفئة الفرعية (سعر شراء/بيع البورصة الأولي) |
| 5 | (اختياري) Trigger للمزامنة عند UPDATE على waste_data_admin | منخفضة | ✅ **منفَّذ** — migration `20260202_trigger_sync_waste_data_admin_to_catalog.sql` يحدّث catalog_waste_materials تلقائياً عند تغيير subcategory_id, name, price_per_kg, weight |
| 6 | سكريبت المزامنة الأولية: المنتجات بدون catalog_waste_id → إنشاء سجل كتالوج + تعيين catalog_waste_id | عالية | ✅ **منفَّذ** — `syncAllProducts()` يتخطى المنتجات المربوطة، وينشئ للباقي سجل كتالوج ويربط catalog_waste_id؛ متاح عبر زر في الصفحة الرئيسية و API `POST /api/admin/sync-catalog` |

## 5. الخلاصة

- **الرأي:** المقترح **جيد جداً** ويمكن اعتماده كسيناريو رسمي.
- **التنفيذ الحالي** يغطي جزءاً كبيراً منه (مزامنة منتج ↔ كتالوج، أسعار الفئات ↔ البورصة، موافقة 10%، حذف آمن).
- **للمطابقة الكاملة مع المقترح:** تكييف أسماء الحقول مع الـ Schema، إضافة `catalog_waste_id` في waste_data_admin، وربط إنشاء المنتج بإنشاء الكتالوج وربط البورصة في نفس التدفق.
- بعد تنفيذ الخطوات ذات الأولوية العالية، **التطبيق سيعمل كما تريد** حسب السيناريو المُحدَّث، مع إمكانية إضافة Trigger وسيناريو «فئة فرعية مع سعر» لاحقاً دون تعارض.

إذا رغبت، يمكن في الخطوة التالية صياغة **ملف migration واحد** (إضافة catalog_waste_id + فهارس) وتعديل **واجهة/خدمة إنشاء المنتج** لتعكس السيناريو 2️⃣ خطوة بخطوة في الكود الحالي.
