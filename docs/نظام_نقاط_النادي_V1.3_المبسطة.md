# نظام نقاط النادي - الإصدار 1.3 المبسّط (Minimal Smart Version)

## 🎯 الهدف من V1.3

إطلاق نسخة أولى **آمنة وبسيطة** من نظام النقاط يمكن تشغيلها سريعًا في المرحلة الأولى، مع:

- أقل عدد ممكن من الجداول والـ Functions
- منطق واضح بدون سياسات معقدة
- سهولة المراجعة والمحاسبة اليدوية
- إمكانية التطوير لاحقًا إلى Version 2.5 بدون كسر النظام

> هذا المستند هو نسخة مبسّطة من تصميم V2.5، ومقصود إنه يكون **“Core ذكي وبسيط”**.

## 1️⃣ المبادئ الأساسية في V1.3

- **نقاط النادي ليست فلوس** → تستخدم فقط في:
  - استبدال جوائز، خصومات، خدمات، سحوبات شهرية.
- **3 حالات فقط للنقاط:**
  - `PENDING` → نقاط مكتسبة لم تُعتمد بعد (تظهر للمستخدم كمعلّقة).
  - `AVAILABLE` → نقاط جاهزة للاستخدام (بعد الاعتماد الشهري).
  - `USED` → نقاط تم استخدامها في جوائز.
- **لا يوجد Expiry / Bonus / Levels / Policies معقدة** في هذه النسخة.
- **التحويل من نقاط المخلفات**:
  - **Admin Only**
  - يدوي
  - بنسبة ثابتة (مثلًا 30%) + سقف شهري بسيط (Hardcoded في الكود أو Function واحدة)
  - ❌ لا يوجد تحويل ذاتي أو تلقائي للمستخدم.
- **الراديو**:
  - يتم تسجيل **Session واحدة** لكل استماع:
    - `start_time`
    - `stop_time`
  - يتم حساب النقاط **مرة واحدة عند الإيقاف** (`EARNED` 1x).
  - ❌ لا نستخدم Checkpoints أو تحديث كل دقيقة في V1.3 (لتقليل الـ Bugs والتلاعب).

### ✅ لغة المنتج الرسمية (مهم جدًا لتفادي خلط المفاهيم)

المنظومة فيها “رصيدين” مختلفين وظيفيًا ونفسيًا، ولازم يتقالوا للمستخدم بنفس الصياغات دائمًا:

#### 🔵 أولاً: رصيد المخلفات (Economic Balance)

- **يتحول كاش فورًا** (من المندوب/الوكيل/الشركة).
- أو **يُستخدم فورًا للشراء في المتجر الإلكتروني**.
- **لا علاقة له بالنادي**.
- **لا يخضع لاعتماد شهري**.

جملة ثابتة مقترحة داخل التطبيق:
- **“رصيد المخلفات هو حقك الاقتصادي الكامل.”**

#### 🟢 ثانيًا: رصيد النادي (Club Rewards Balance)

- **مكافآت/جوائز/سحوبات/خصومات** (قيمة اجتماعية وتحفيزية).
- **يُعتمد شهريًا** (رصيد هذا الشهر → رصيد متاح).
- **لا يتحول كاش**.
- **لا يُستخدم في المتجر النقدي**.

جملة ثابتة مقترحة داخل التطبيق:
- **“نقاط النادي نظام مكافآت وتحفيز، وليست مقابل مالي.”**

> **ملاحظة UX مهمة:** داخل واجهة المستخدم، تجنّب كلمة “تحويل” بين الرصيدين. استخدم: “مكافأة”، “تقدير”، “رصيد تحفيزي”.

## 2️⃣ نطاق التنفيذ في V1.3

### أ. الجداول المطلوبة (Minimal Core)

ينفَّذ الآن (إجباري):

1. **`club_service_points_config` (مبسّطة)**
   - هدفها: تعريف معدل النقاط للخدمات الأساسية فقط.
   - الحقول الأساسية المقترحة:

   ```sql
   CREATE TABLE IF NOT EXISTS club_service_points_config (
     id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
     service_name VARCHAR(100) NOT NULL UNIQUE,  -- 'radio_stream', 'manual_admin_add', ...
     service_title VARCHAR(255) NOT NULL,
     service_type VARCHAR(50) NOT NULL CHECK (service_type IN (
       'radio',
       'manual',
       'other'
     )),
     points_rate_type VARCHAR(20) NOT NULL CHECK (points_rate_type IN (
       'per_minute',    -- للراديو
       'fixed'          -- لإضافة/خصم يدوي
     )),
     base_points_rate DECIMAL(10, 2) NOT NULL DEFAULT 1.0, -- مثال: 1 نقطة/دقيقة
     is_active BOOLEAN DEFAULT TRUE,
     created_at TIMESTAMPTZ DEFAULT NOW(),
     updated_at TIMESTAMPTZ DEFAULT NOW()
   );
   ```

   - لا نستخدم الآن:
     - `multiplier_factor`
     - `bonus_conditions`
     - `max_points_per_day`
     - أي JSONB معقّد.

2. **`club_points_wallet`**

   - الحقول الأساسية:

   ```sql
   CREATE TABLE IF NOT EXISTS club_points_wallet (
     id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
     user_id UUID NOT NULL UNIQUE REFERENCES new_profiles(id),
     pending_points INTEGER NOT NULL DEFAULT 0,
     available_points INTEGER NOT NULL DEFAULT 0,
     used_points INTEGER NOT NULL DEFAULT 0,
     lifetime_points INTEGER NOT NULL DEFAULT 0,
     created_at TIMESTAMPTZ DEFAULT NOW(),
     updated_at TIMESTAMPTZ DEFAULT NOW()
   );
   ```

   - التعريف:
     - `pending_points`: ما تم كسبه وينتظر الاعتماد.
     - `available_points`: الرصيد الجاهز للاستخدام.
     - `used_points`: ما تم استخدامه في جوائز.
     - `lifetime_points`: إجمالي النقاط المكتسبة (لأغراض التقارير).

3. **`club_points_transactions`** (نسخة بسيطة)

   ```sql
   CREATE TABLE IF NOT EXISTS club_points_transactions (
     id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
     user_id UUID NOT NULL REFERENCES new_profiles(id),
     transaction_type VARCHAR(20) NOT NULL CHECK (transaction_type IN (
       'EARNED',     -- كسب نقاط (تذهب إلى PENDING)
       'ACTIVATED',  -- اعتماد نقاط (PENDING → AVAILABLE)
       'USED'        -- استخدام نقاط (AVAILABLE → USED)
     )),
     points INTEGER NOT NULL,          -- يمكن أن يكون موجب أو سالب في USED
     points_before INTEGER NOT NULL,   -- رصيد AVAILABLE قبل العملية
     points_after INTEGER NOT NULL,    -- رصيد AVAILABLE بعد العملية
     source VARCHAR(100),              -- 'radio_stream', 'admin_manual', 'waste_conversion'
     reference_id UUID,               -- جلسة راديو، طلب تحويل، استبدال جائزة، إلخ
     description TEXT,
     created_by UUID REFERENCES new_profiles(id),
     created_at TIMESTAMPTZ DEFAULT NOW()
   );
   ```

   - في V1.3:
     - **لا نستخدم**: `BONUS`, `ADJUSTED`, `EXPIRED`, `CONVERTED` كـ transaction_types مستقلة.
     - التحويل من المخلفات يُسجَّل أيضًا تحت نوع `EARNED` ولكن مصدره `waste_conversion` (مع مرجع واضح).

4. **`club_rewards`** (كتالوج الجوائز)

   ```sql
   CREATE TABLE IF NOT EXISTS club_rewards (
     id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
     title VARCHAR(255) NOT NULL,
     description TEXT,
     points_required INTEGER NOT NULL,
     is_active BOOLEAN DEFAULT TRUE,
     stock INTEGER, -- NULL = غير محدود
     created_at TIMESTAMPTZ DEFAULT NOW(),
     updated_at TIMESTAMPTZ DEFAULT NOW()
   );
   ```

5. **`club_reward_redemptions`** (استبدال الجوائز)

   ```sql
   CREATE TABLE IF NOT EXISTS club_reward_redemptions (
     id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
     user_id UUID NOT NULL REFERENCES new_profiles(id),
     reward_id UUID NOT NULL REFERENCES club_rewards(id),
     points_spent INTEGER NOT NULL,
     status VARCHAR(20) NOT NULL DEFAULT 'REQUESTED' CHECK (status IN (
       'REQUESTED',
       'APPROVED',
       'REJECTED',
       'DELIVERED'
     )),
     notes TEXT,
     created_at TIMESTAMPTZ DEFAULT NOW(),
     updated_at TIMESTAMPTZ DEFAULT NOW()
   );
   ```

   - في V1.3:
     - **طريقة الاستلام** يمكن كتابتها في `notes` أو حقل نصي بسيط (لا نحتاج الآن `collection_points` معقّد).

6. **`monthly_points_settlement`** (اعتماد شهري يدوي)

   ```sql
   CREATE TABLE IF NOT EXISTS monthly_points_settlement (
     id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
     settlement_month VARCHAR(7) NOT NULL, -- '2026-01'
     total_pending_points INTEGER NOT NULL DEFAULT 0,
     total_activated_points INTEGER NOT NULL DEFAULT 0,
     total_users_affected INTEGER NOT NULL DEFAULT 0,
     processed_by UUID REFERENCES new_profiles(id),
     processed_at TIMESTAMPTZ DEFAULT NOW(),
     status VARCHAR(20) NOT NULL DEFAULT 'completed'
       CHECK (status IN ('completed')),
     notes TEXT,
     UNIQUE(settlement_month)
   );
   ```

   - في V1.3:
     - **لا نحتاج حالات**: `pending/failed/processing`، لأن الاعتماد سيكون عن طريق زر واحد وبكميات صغيرة في البداية.

❌ مؤجَّل رسميًا (Stage 2 / V2.5):

- `collection_points` (ممكن حاليًا اعتبار منفذ الاستلام = الجمعية وكتابة ذلك في `notes`).
- `club_points_policies` (سياسات JSONB معقّدة).
- أي Approval Workflow متقدّم.
- أي Expiry للنقاط.

## 3️⃣ Functions المطلوبة في V1.3

في هذه النسخة ننفذ فقط 4 Functions أساسية:

### 1. `calculate_service_points` (مستخدمة أساسًا مع الراديو)

فكرتها كما في المستند السابق، لكن نستخدمها فقط مع:

- خدمة واحدة أساسية الآن: `radio_stream`
- نوع معدل: `per_minute`

يمكن تبسيطها إلى:

```sql
CREATE OR REPLACE FUNCTION calculate_service_points(
  p_service_name VARCHAR,
  p_minutes DECIMAL
) RETURNS INTEGER
LANGUAGE plpgsql
STABLE
AS $$
DECLARE
  v_rate DECIMAL(10, 2);
  v_points INTEGER;
BEGIN
  SELECT base_points_rate
  INTO v_rate
  FROM club_service_points_config
  WHERE service_name = p_service_name
    AND is_active = TRUE;

  IF v_rate IS NULL THEN
    -- افتراضي: 1 نقطة/دقيقة
    v_rate := 1.0;
  END IF;

  v_points := FLOOR(p_minutes * v_rate);
  RETURN GREATEST(v_points, 0);
END;
$$;
```

> لاحقًا يمكن تطويرها لدعم أنواع أخرى (`per_action`, `fixed`)، لكن الآن نركز على الراديو فقط.

---

### 2. `update_club_points_wallet` (كما في النسخة المصححة المبسّطة)

نستخدم النسخة المصححة في المستند السابق، لكن في V1.3:

- **أنواع المعاملات المستخدمة فعليًا**: `EARNED`, `ACTIVATED`, `USED`.
- لا نستخدم المنطق المعقّد لـ `BONUS`, `ADJUSTED`, إلخ.

(الكود نفسه كما في المستند السابق بعد التصحيح، مع تجاهل الفروع غير المستخدمة الآن.)

---

### 3. `activate_monthly_points` (اعتماد شهري يدوي وبسيط)

تُستخدم كما تم تعديلها:

- Loop على كل مستخدم لديه `pending_points > 0`.
- استدعاء `update_club_points_wallet(..., 'ACTIVATED', ...)` لكل مستخدم.
- تسجيل إجمالي ما تم اعتماده في `monthly_points_settlement`.

> لا توجد سياسات معقّدة هنا (مثل نسب مختلفة أو استثناءات)، مجرد “اعتماد كل ما هو PENDING”.

---

### 4. `get_user_points_summary`

كما في المستند السابق، مع تبسيط:

```sql
CREATE OR REPLACE FUNCTION get_user_points_summary(p_user_id UUID)
RETURNS TABLE (
  pending_points INTEGER,
  available_points INTEGER,
  used_points INTEGER,
  total_balance INTEGER,
  lifetime_points INTEGER
)
LANGUAGE sql
STABLE
AS $$
  SELECT 
    COALESCE(pending_points, 0),
    COALESCE(available_points, 0),
    COALESCE(used_points, 0),
    COALESCE(available_points, 0),  -- الرصيد الجاهز للاستخدام
    COALESCE(lifetime_points, 0)
  FROM club_points_wallet
  WHERE user_id = p_user_id;
$$;
```

## 4️⃣ منطق الراديو في V1.3 (Session-Based)

### الفكرة الأساسية

- لكل مستمع جلسة واحدة أو أكثر:
  - `radio_session` (جدول أو تمثيل منطقي في `radio_listeners`):
    - `id`
    - `user_id`
    - `stream_id`
    - `start_time`
    - `stop_time`
- عند الضغط على **تشغيل**:
  - يتم تسجيل `start_time`.
- عند الضغط على **إيقاف** أو انتهاء البث:
  - يتم حساب مدة الاستماع بالدقائق.
  - يتم استدعاء:
    - `calculate_service_points('radio_stream', minutes_listened)`
    - ثم `update_club_points_wallet(user_id, points, 'EARNED', ..., 'radio_stream', session_id)`

### لماذا هذا أبسط وأأمن من Checkpoints؟

- لا يوجد:
  - تحديث كل دقيقة.
  - مشاكل Network / Refresh / Duplicate Events.
- كل Session لها عملية حساب واحدة:
  - سهل تتبّعها في التقارير.
  - سهل مراجعتها يدويًا لو حدثت مشكلة.

## 5️⃣ التحويل من نقاط المخلفات في V1.3 (Manual Admin Only)

### القواعد في V1.3

- التحويل يتم **فقط** بواسطة الأدمن (من لوحة التحكم).
- نسبة التحويل:
  - ثابتة في الكود، مثلًا: `30%`.
- سقف شهري بسيط:
  - مثلًا: `max 2000 نقطة نادي لكل مستخدم في الشهر`.
  - يمكن تخزينه في إعداد backend عادي أو constant في الكود.
- تسجيل التحويل:
  - في `club_points_transactions` كمعاملة `EARNED` مع:
    - `source = 'waste_conversion'`
    - `description` واضحة (مثال: "تحويل 100 نقطة مخلفات إلى 30 نقطة نادي").

### لماذا نلغي التحويل الذاتي في V1.3؟

- حماية منظومة الجوائز من:
  - استنزاف مفاجئ.
  - سوء استخدام.
- إعطاء فرصة:
  - لاختبار النموذج المالي.
  - لضبط النسب قبل فتح التحويل للمستخدمين أنفسهم في V2.5.

## ⚠️ المخاطر ونقاط الانتباه في V1.3 (بصراحة)

مفيش نظام كامل بدون نقاط انتباه. دي أهم 3 مخاطر واقعية في V1.3 + الحلول التشغيلية المقترحة:

### ⚠️ 1. العامل النفسي للمستخدم (الاعتماد الشهري قد يسبب إحباط)

**السيناريو المتوقع:**
- المستخدم يقول: **“يعني أنا سمعت الراديو ومش هعرف أستبدل دلوقتي؟”**

**الخطر:**
- لو اللغة مش واضحة، المستخدم هيفهم النقاط كأنها “فلوس محبوسة”.

**الحل (تشغيلي/UX وليس تقني):**
- **تسمية واضحة داخل شاشة النقاط**:
  - **“رصيد هذا الشهر”** = `PENDING`
  - **“رصيد متاح”** = `AVAILABLE`
- **Pop-up / Tooltip بسيط**:
  - **“النقاط تُعتمد نهاية الشهر للحفاظ على استدامة الجوائز.”**
- **إظهار تاريخ الاعتماد القادم** (مثلاً: “الاعتماد القادم: 1 فبراير”).

### ⚠️ 2. الاعتماد الشهري = مسؤولية تشغيلية (زر واحد لكنه يحتاج التزام)

**الخطر:**
- لو الاعتماد اتأخر، `PENDING` هتتراكم، والمستخدم يحس إن في مشكلة أو ظلم.

**الحل (تشغيلي):**
- تحديد يوم ثابت مُعلن رسميًا:
  - **“اعتماد النقاط يوم 1 من كل شهر.”**
- تعيين “مسؤول اعتماد” داخل الفريق (اسم/دور).
- تنبيه داخل لوحة الأدمن لو لم يتم الاعتماد لهذا الشهر بعد.

### ⚠️ 3. كثرة Manual Admin Add (تعب + أخطاء بشرية + تضخم غير منضبط)

**الخطر:**
- لو إضافة النقاط اليدوية بقت الاستخدام اليومي الأساسي:
  - هتزيد أخطاء الإدخال.
  - هيتعب الفريق.
  - وهتفتح باب اختلافات غير منضبطة في منح النقاط.

**الحل (تشغيلي):**
- استخدمها فقط في:
  - **مكافآت محددة**
  - **حملات/فعاليات**
  - **تصحيح أخطاء نادرة**
- عدم استخدامها كبديل دائم للأتمتة (اللي هتيجي في V2.5).
- تشغيليًا: إلزام وصف واضح في كل عملية إضافة/خصم (داخل `description`) حتى لو قاعدة البيانات لا تفرضه في V1.3.

## 🧩 Critical Execution Notes (تنفيذ V1.3 على أرض الواقع بدون تعقيد)

هذا القسم يوضّح نقاط الانتباه العملية عند تنفيذ V1.3، مع حلول **لا تُعقّد النظام** ولا ترفعه إلى V2.5، لكنها تحمي التشغيل وتجربة المستخدم.

### 1️⃣ التحويل اليدوي الكامل = عنق زجاجة تشغيلي 🚨

**المشكلة:**
- التحويل من نقاط المخلفات إلى نقاط النادي في V1.3 هو:
  - Admin Only
  - يدوي بالكامل
- مع 50–100 مستخدم نشط شهريًا: (مراجعة/إدخال/استفسارات) → ضغط على الأدمن + تأخير + أخطاء بشرية.

**الحل المقترح داخل V1.3 (بدون تحويل ذاتي/تلقائي):**
- إضافة **جدول طلبات تحويل بسيط**: المستخدم “يطلب فقط”، والأدمن “يوافق/يرفض”، والتنفيذ يظل تحت السيطرة.

```sql
CREATE TABLE IF NOT EXISTS recycling_conversion_requests (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES new_profiles(id),
  recycling_points INTEGER NOT NULL,
  club_points_expected INTEGER NOT NULL, -- حساب ثابت (مثلاً 30%)
  status VARCHAR(20) NOT NULL DEFAULT 'PENDING' CHECK (status IN (
    'PENDING',
    'APPROVED',
    'REJECTED'
  )),
  processed_by UUID REFERENCES new_profiles(id),
  processed_at TIMESTAMPTZ,
  rejection_reason TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

**آلية العمل:**
- المستخدم: ينشئ Request فقط (لا خصم/لا إضافة نقاط).
- الأدمن: يراجع ويضغط Approve/Reject.
- عند Approve:
  - خصم نقاط المخلفات يتم في نظام المخلفات (منفصل).
  - يتم تسجيل معاملة نادي:
    - `club_points_transactions`: `transaction_type='EARNED'`, `source='waste_conversion'`
    - وتحديث `club_points_wallet` عبر `update_club_points_wallet(...)`

**ملاحظة مهمة:**
- هذا **ليس تحويلًا ذاتيًا تلقائيًا**؛ هو تنظيم لعملية يدوية لتقليل الضغط وإضافة سجل مراجعة واضح.

### 2️⃣ خطر نسيان الاعتماد الشهري 📅

**المشكلة:**
- V1.3 يعتمد على اعتماد شهري يدوي (زر واحد).
- الخطر: نسيان الاعتماد → تراكم `pending_points` → إحباط المستخدمين.

**الحل داخل V1.3 (تذكير فقط – بدون Workflow/Automation):**
- Function بسيطة تفحص هل اعتماد الشهر الحالي تم أم لا، وتستخدمها لوحة الأدمن لعرض Banner تحذيري.

```sql
CREATE OR REPLACE FUNCTION check_monthly_settlement_due()
RETURNS BOOLEAN
LANGUAGE plpgsql
AS $$
DECLARE
  v_current_month VARCHAR(7);
  v_last_settlement VARCHAR(7);
BEGIN
  v_current_month := TO_CHAR(NOW(), 'YYYY-MM');

  SELECT settlement_month INTO v_last_settlement
  FROM monthly_points_settlement
  ORDER BY settlement_month DESC
  LIMIT 1;

  IF v_last_settlement IS NULL OR v_last_settlement < v_current_month THEN
    RETURN TRUE; -- مطلوب اعتماد
  END IF;

  RETURN FALSE;
END;
$$;
```

**في لوحة التحكم:**
- Banner واضح:
  - **“⚠️ اعتماد نقاط الشهر الحالي مطلوب”**

### 3️⃣ الراديو Session-Based ممتاز… لكن انتبه للجلسات المتروكة 🎵

**السيناريو الواقعي:**
- المستخدم يبدأ الاستماع.
- التطبيق يُغلق / النت ينقطع.
- لا يتم تسجيل `stop_time` → Session تظل مفتوحة → النقاط لا تُحسب.

**الحل داخل V1.3: Auto-close للجلسات المتروكة (بدون checkpoints):**

```sql
CREATE OR REPLACE FUNCTION close_abandoned_radio_sessions()
RETURNS INTEGER
LANGUAGE plpgsql
AS $$
DECLARE
  v_closed_count INTEGER;
BEGIN
  UPDATE radio_listeners
  SET
    stop_time = start_time + INTERVAL '4 hours',
    updated_at = NOW()
  WHERE stop_time IS NULL
    AND start_time < NOW() - INTERVAL '4 hours';

  GET DIAGNOSTICS v_closed_count = ROW_COUNT;
  RETURN v_closed_count;
END;
$$;
```

**متى تُستدعى؟**
- يوميًا (Cron) أو قبل عملية حساب/تسوية النقاط.
