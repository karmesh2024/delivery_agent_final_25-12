# وثائق سياسات RLS لإدارة المجموعات

يهدف هذا المستند إلى توثيق سياسات الأمان على مستوى الصف (Row Level Security - RLS) ودالة SQL المساعدة التي تم إنشاؤها للتحكم في الوصول إلى جدولي `admin_groups` و `admin_group_members` في قاعدة بيانات Supabase. تعتمد هذه السياسات على دور المستخدم المصادق عليه حاليًا.

## 1. الدالة المساعدة: `public.get_my_role()`

تم إنشاء هذه الدالة لتحديد رمز الدور (Role Code) للمستخدم المصادق عليه حاليًا. تُستخدم هذه الدالة داخل سياسات RLS للتحقق من صلاحيات المستخدم.

**الغرض:**
- جلب معرف المستخدم الحالي من Supabase Auth (`auth.uid()`).
- البحث عن هذا المستخدم في جدول `public.admins`.
- ربط `role_id` من جدول `admins` بجدول `public.roles`.
- إرجاع قيمة عمود `code` من جدول `roles` المقابل لدور المستخدم.
- التأكد من أن المستخدم نشط (`is_active = true`).

**كود SQL:**

```sql
-- حذف الدالة القديمة إذا كانت موجودة لتجنب الأخطاء
DROP FUNCTION IF EXISTS public.get_my_role();

-- إنشاء الدالة المصححة
CREATE OR REPLACE FUNCTION public.get_my_role()
RETURNS TEXT -- نوع البيانات الذي يعيده (عادةً رمز الدور)
LANGUAGE sql
STABLE -- تعني أن الدالة لا تعدل البيانات وتعيد نفس النتيجة لنفس المدخلات ضمن استعلام واحد
AS $$
  SELECT r.code -- <-- افترضنا وجود عمود 'code' في جدول 'roles'
  FROM public.admins a
  JOIN public.roles r ON a.role_id = r.id -- الربط مع جدول roles
  WHERE a.user_id = auth.uid() AND a.is_active = true; -- التحقق من أن المستخدم نشط أيضًا
$$;

-- منح الإذن للمستخدمين المصادق عليهم لاستدعاء الدالة
GRANT EXECUTE ON FUNCTION public.get_my_role() TO authenticated;
```

**المتطلبات:**
- وجود جدول `public.admins` مع عمود `user_id` (مرتبط بـ `auth.users.id`) وعمود `role_id` وعمود `is_active`.
- وجود جدول `public.roles` مع عمود `id` وعمود `code` (أو العمود الذي يتم استخدامه لتخزين رمز الدور النصي).

## 2. سياسات RLS لجدول `admin_groups`

تم تفعيل RLS على جدول `admin_groups` وإنشاء السياسات التالية للتحكم في الوصول:

**أ. سياسة القراءة (SELECT):**
- **الاسم:** `Allow authenticated read access to admin groups`
- **الغرض:** السماح لأي مستخدم قام بتسجيل الدخول بقراءة بيانات جميع المجموعات.
- **كود SQL:**
  ```sql
  CREATE POLICY "Allow authenticated read access to admin groups"
  ON public.admin_groups
  FOR SELECT
  TO authenticated
  USING (true);
  ```
- **شرح:** `USING (true)` يعني عدم وجود قيود على مستوى الصف للقراءة.

**ب. سياسة الإضافة (INSERT):**
- **الاسم:** `Allow admins to insert admin groups`
- **الغرض:** السماح فقط للمستخدمين الذين لديهم دور 'admin' أو 'super_admin' بإضافة مجموعات جديدة.
- **كود SQL:**
  ```sql
  CREATE POLICY "Allow admins to insert admin groups"
  ON public.admin_groups
  FOR INSERT
  TO authenticated
  WITH CHECK (public.get_my_role() IN ('admin', 'super_admin'));
  ```
- **شرح:** `WITH CHECK` تستدعي `get_my_role()` للتحقق من أن دور المستخدم الحالي هو 'admin' أو 'super_admin' قبل السماح بعملية الإضافة.

**ج. سياسة التعديل (UPDATE):**
- **الاسم:** `Allow admins to update admin groups`
- **الغرض:** السماح فقط للمستخدمين الذين لديهم دور 'admin' أو 'super_admin' بتعديل بيانات المجموعات الموجودة.
- **كود SQL:**
  ```sql
  CREATE POLICY "Allow admins to update admin groups"
  ON public.admin_groups
  FOR UPDATE
  TO authenticated
  USING (public.get_my_role() IN ('admin', 'super_admin'))
  WITH CHECK (public.get_my_role() IN ('admin', 'super_admin'));
  ```
- **شرح:** `USING` تحدد الصفوف التي يمكن للمستخدم تعديلها (بناءً على الدور)، و `WITH CHECK` تضيف تحققًا إضافيًا على الدور عند محاولة التعديل.

**د. سياسة الحذف (DELETE):**
- **الاسم:** `Allow admins to delete admin groups`
- **الغرض:** السماح فقط للمستخدمين الذين لديهم دور 'admin' أو 'super_admin' بحذف المجموعات.
- **كود SQL:**
  ```sql
  CREATE POLICY "Allow admins to delete admin groups"
  ON public.admin_groups
  FOR DELETE
  TO authenticated
  USING (public.get_my_role() IN ('admin', 'super_admin'));
  ```
- **شرح:** `USING` تحدد الصفوف التي يمكن للمستخدم حذفها بناءً على دوره.

## 3. سياسات RLS لجدول `admin_group_members`

تم تفعيل RLS على جدول `admin_group_members` (جدول الربط) وإنشاء السياسات التالية:

**أ. سياسة القراءة (SELECT):**
- **الاسم:** `Allow authenticated read access to group memberships`
- **الغرض:** السماح لأي مستخدم قام بتسجيل الدخول بقراءة علاقات عضوية المجموعات (من ينتمي لأي مجموعة).
- **كود SQL:**
  ```sql
  CREATE POLICY "Allow authenticated read access to group memberships"
  ON public.admin_group_members
  FOR SELECT
  TO authenticated
  USING (true);
  ```

**ب. سياسة الإضافة (INSERT):**
- **الاسم:** `Allow admins to insert group members`
- **الغرض:** السماح فقط للمستخدمين الذين لديهم دور 'admin' أو 'super_admin' بإضافة أعضاء جدد إلى المجموعات.
- **كود SQL:**
  ```sql
  CREATE POLICY "Allow admins to insert group members"
  ON public.admin_group_members
  FOR INSERT
  TO authenticated
  WITH CHECK (public.get_my_role() IN ('admin', 'super_admin'));
  ```

**ج. سياسة الحذف (DELETE):**
- **الاسم:** `Allow admins to delete group members`
- **الغرض:** السماح فقط للمستخدمين الذين لديهم دور 'admin' أو 'super_admin' بحذف أعضاء من المجموعات.
- **كود SQL:**
  ```sql
  CREATE POLICY "Allow admins to delete group members"
  ON public.admin_group_members
  FOR DELETE
  TO authenticated
  USING (public.get_my_role() IN ('admin', 'super_admin'));
  ```

## 4. ملاحظات هامة

- **التخصيص:** هذه السياسات هي نقطة بداية. قد تحتاج إلى تعديل الأدوار المسموح بها (`'admin', 'super_admin'`) أو إضافة شروط أكثر تحديدًا (مثل السماح لمدير المجموعة فقط بتعديلها أو إضافة أعضاء) بناءً على متطلبات الأمان الدقيقة لتطبيقك.
- **الاختبار:** من الضروري اختبار هذه السياسات جيدًا باستخدام حسابات مستخدمين بأدوار مختلفة للتأكد من أنها تعمل كما هو متوقع وتمنع الوصول غير المصرح به.
- **دالة `get_my_role()`:** تأكد من أن دالة `get_my_role()` تعمل بشكل صحيح وتعكس بدقة كيفية تخزين الأدوار في قاعدة البيانات لديك.
