# نظام إدارة الصلاحيات والأدوار

## مقدمة

نظام الصلاحيات هو جزء أساسي من أي نظام حديث لإدارة الوصول والتحكم في الموارد. يعتمد النظام على ثلاثة جداول رئيسية تعمل معًا لتوفير إدارة مرنة وقوية للصلاحيات:

- `roles`: جدول الأدوار 
- `permissions`: جدول الصلاحيات
- `role_permissions`: جدول العلاقات بين الأدوار والصلاحيات

## العلاقة بين الجداول

### نموذج العلاقة (ER Diagram)

```
+---------------+       +---------------------+       +------------------+
|    roles      |       |  role_permissions   |       |   permissions    |
+---------------+       +---------------------+       +------------------+
| id (PK)       |<----->| role_id (FK)        |<----->| id (PK)          |
| name          |       | permission_id (FK)  |       | resource_id (FK) |
| description   |       | created_at          |       | action_id (FK)   |
| level         |       | created_by          |       | description      |
| is_system     |       |                     |       | code             |
| is_active     |       |                     |       | created_at       |
| permissions   |       |                     |       | name             |
| ...           |       |                     |       | ...              |
+---------------+       +---------------------+       +------------------+
```

### شرح العلاقة Many-to-Many (علاقة متعددة لمتعددة)

العلاقة بين الأدوار والصلاحيات هي علاقة **متعددة لمتعددة** (Many-to-Many):

- **دور واحد** يمكن أن يملك **عدة صلاحيات**
- **صلاحية واحدة** يمكن أن تكون موجودة في **عدة أدوار**

لتنفيذ هذه العلاقة في قواعد البيانات العلائقية، نحتاج إلى **جدول وسيط** (Junction Table) وهو `role_permissions` في هذه الحالة.

## جدول `role_permissions` كجدول وسيط

### تعريف الجدول

```sql
CREATE TABLE IF NOT EXISTS public.role_permissions (
  id uuid NOT NULL DEFAULT extensions.uuid_generate_v4(),
  role_id uuid NOT NULL REFERENCES public.roles(id) ON DELETE CASCADE,
  permission_id uuid NOT NULL REFERENCES public.permissions(id) ON DELETE CASCADE,
  created_at timestamptz NOT NULL DEFAULT CURRENT_TIMESTAMP,
  created_by uuid, -- من أضاف هذه الصلاحية للدور
  
  CONSTRAINT role_permissions_pkey PRIMARY KEY (id),
  CONSTRAINT role_permissions_role_permission_unique UNIQUE (role_id, permission_id)
);
```

### كيف يعمل الجدول الوسيط؟

1. **تخزين العلاقات**: كل سجل في جدول `role_permissions` يمثل علاقة بين دور محدد وصلاحية محددة.

2. **المفاتيح الأجنبية**: 
   - `role_id`: مفتاح أجنبي يشير إلى جدول `roles`
   - `permission_id`: مفتاح أجنبي يشير إلى جدول `permissions`

3. **ضمان تفرد العلاقات**: القيد `role_permissions_role_permission_unique` يضمن عدم تكرار نفس العلاقة بين الدور والصلاحية.

4. **التنظيف التلقائي**: باستخدام `ON DELETE CASCADE`، عند حذف دور أو صلاحية يتم تلقائيًا حذف جميع العلاقات المرتبطة بهما.

## مثال عملي للعلاقات

### مثال 1: دور "مدير النظام" له عدة صلاحيات

لنفترض أن لدينا دور "مدير النظام" بمعرف `00000000-0000-0000-0000-000000000001` ونريد منحه صلاحيات متعددة:

```sql

-- إضافة العلاقات في جدول role_permissions
INSERT INTO role_permissions (role_id, permission_id) VALUES

```

لنفترض أن صلاحية "عرض الطلبات" بمعرف `aaee9f5c-e74c-47e8-8d9e-18e18d7a75d9` يجب أن تكون متاحة لعدة أدوار:

```sql

```

## دالة `setup_permissions` لإدارة العلاقات

النظام يستخدم دالة `setup_permissions` لإنشاء وإدارة العلاقات بين الأدوار والصلاحيات بشكل تلقائي:

```sql
CREATE OR REPLACE FUNCTION setup_permissions()
RETURNS void AS $$
DECLARE
    resource_rec RECORD;
    action_rec RECORD;
    permission_id uuid;
    role_rec RECORD;
BEGIN
    -- إنشاء الصلاحيات من تركيبة الموارد والإجراءات
    FOR resource_rec IN SELECT * FROM public.resources WHERE is_active = true LOOP
        FOR action_rec IN SELECT * FROM public.actions LOOP
            -- إنشاء رمز مركب
            INSERT INTO public.permissions (resource_id, action_id, description, code)
            VALUES (
                resource_rec.id,
                action_rec.id,
                action_rec.name || ' ' || resource_rec.name,
                resource_rec.code || ':' || action_rec.code
            )
            ON CONFLICT (resource_id, action_id) DO UPDATE
            SET description = EXCLUDED.description, code = EXCLUDED.code
            RETURNING id INTO permission_id;
            
            -- منح الصلاحية لدور super_admin
            IF EXISTS (SELECT 1 FROM public.roles WHERE id = '00000000-0000-0000-0000-000000000001') THEN
                INSERT INTO public.role_permissions (role_id, permission_id)
                VALUES ('00000000-0000-0000-0000-000000000001', permission_id)
                ON CONFLICT (role_id, permission_id) DO NOTHING;
            END IF;
            
            -- منح صلاحيات عرض لدور viewer
            IF action_rec.code = 'view' AND EXISTS (SELECT 1 FROM public.roles WHERE id = '00000000-0000-0000-0000-000000000006') THEN
                INSERT INTO public.role_permissions (role_id, permission_id)
                VALUES ('00000000-0000-0000-0000-000000000006', permission_id)
                ON CONFLICT (role_id, permission_id) DO NOTHING;
            END IF;
            
            -- منح صلاحيات إدارة للمدراء
            IF action_rec.code IN ('view', 'create', 'update', 'toggle', 'export', 'assign', 'approve') 
               AND resource_rec.code != 'admins' 
               AND resource_rec.code != 'roles' 
               AND resource_rec.code != 'settings'
               AND EXISTS (SELECT 1 FROM public.roles WHERE id = '00000000-0000-0000-0000-000000000002') THEN
                INSERT INTO public.role_permissions (role_id, permission_id)
                VALUES ('00000000-0000-0000-0000-000000000002', permission_id)
                ON CONFLICT (role_id, permission_id) DO NOTHING;
            END IF;
        END LOOP;
    END LOOP;
    
    -- تحديث جدول admins لتحويل الصلاحيات من jsonb إلى علاقات
    FOR role_rec IN SELECT * FROM public.roles LOOP
        UPDATE public.admins
        SET role_id = role_rec.id
        WHERE (permissions->>'role') = role_rec.name AND role_id IS NULL;
    END LOOP;
END;
$$ LANGUAGE plpgsql;
```

## كيفية التعامل مع حقل `permissions` الموجود في جدول `roles`

في تصميم النظام الحالي، يحتوي جدول `roles` على حقل `permissions` من نوع JSONB، ويمكن اعتباره:

1. **للتوافق مع النسخة السابقة**: للحفاظ على التوافق مع الأنظمة القديمة.
2. **للصلاحيات الاستثنائية**: تخزين صلاحيات خاصة أو مؤقتة لا تتطلب تسجيلاً دائماً.
3. **للاستخدامات المتقدمة**: تخزين إعدادات إضافية متعلقة بالصلاحيات مثل النطاق أو القيود.

لكن التوصية هي الاعتماد بشكل أساسي على العلاقات في جدول `role_permissions` لإدارة الصلاحيات.

## الانتقال من JSONB إلى العلاقات العلائقية

يمكن إضافة مشغّل (trigger) لتحويل الصلاحيات من حقل JSONB إلى علاقات في جدول `role_permissions` عند إنشاء أو تحديث دور:

```sql
CREATE OR REPLACE FUNCTION sync_role_permissions()
RETURNS TRIGGER AS $$
DECLARE
    perm_key TEXT;
    perm_value BOOLEAN;
    permission_id UUID;
BEGIN
    -- إذا تم تحديد صلاحيات في حقل JSONB
    IF NEW.permissions IS NOT NULL AND jsonb_typeof(NEW.permissions) = 'object' THEN
        -- المرور على كل صلاحية في الكائن JSONB
        FOR perm_key, perm_value IN SELECT * FROM jsonb_each_text(NEW.permissions) LOOP
            -- التحقق من قيمة الصلاحية (يجب أن تكون true لإضافتها)
            IF perm_value::BOOLEAN = true THEN
                -- البحث عن معرف الصلاحية
                SELECT id INTO permission_id FROM public.permissions WHERE code = perm_key;
                
                -- إذا وجدت الصلاحية، أضفها إلى جدول العلاقات
                IF permission_id IS NOT NULL THEN
                    INSERT INTO public.role_permissions (role_id, permission_id)
                    VALUES (NEW.id, permission_id)
                    ON CONFLICT (role_id, permission_id) DO NOTHING;
                END IF;
            ELSE
                -- إذا كانت الصلاحية false، احذفها من جدول العلاقات
                DELETE FROM public.role_permissions 
                WHERE role_id = NEW.id 
                AND permission_id IN (SELECT id FROM public.permissions WHERE code = perm_key);
            END IF;
        END LOOP;
    END IF;
    
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- إنشاء المشغّل
CREATE TRIGGER sync_role_permissions_trigger
AFTER INSERT OR UPDATE ON public.roles
FOR EACH ROW
EXECUTE FUNCTION sync_role_permissions();
```

## فوائد استخدام جدول الوسيط `role_permissions`

### 1. سلامة البيانات

- **المفاتيح الأجنبية**: تضمن أن جميع العلاقات تشير إلى كيانات موجودة فعلاً.
- **تفرد العلاقات**: يتم ضمان عدم تكرار العلاقة بين نفس الدور والصلاحية.
- **التحقق التلقائي**: النظام العلائقي يتحقق تلقائيًا من صحة جميع العلاقات.

### 2. أداء أفضل للاستعلامات المعقدة

- **استعلامات JOIN**: تتيح استعلامات JOIN أكثر كفاءة وأداء مقارنة بالبحث في هياكل JSONB.
- **الفهارس**: يمكن إنشاء فهارس على أعمدة جدول العلاقات لتسريع عمليات البحث.
- **استعلامات تحليلية**: تسمح بإنشاء استعلامات تحليلية أكثر تعقيدًا على العلاقات.

### 3. سهولة الإدارة والصيانة

- **مركزية التحكم**: إدارة جميع العلاقات من مكان واحد.
- **تتبع التغييرات**: يمكن تتبع من قام بإضافة صلاحية ومتى (من خلال حقل created_by و created_at).
- **آلية موحدة**: نفس آلية التعامل مع جميع أنواع العلاقات في النظام.

## البحث عن صلاحيات المستخدم

دالة `check_admin_permission` تستفيد من هيكل العلاقات لاختبار ما إذا كان للمستخدم صلاحية معينة:

```sql
CREATE OR REPLACE FUNCTION check_admin_permission(admin_id uuid, permission_code text)
RETURNS boolean AS $$
DECLARE
    has_permission boolean;
BEGIN
    -- التحقق من وجود الصلاحية مباشرة من خلال الدور
    SELECT EXISTS (
        SELECT 1 
        FROM public.admins a
        JOIN public.role_permissions rp ON a.role_id = rp.role_id
        JOIN public.permissions p ON rp.permission_id = p.id
        WHERE a.id = admin_id AND p.code = permission_code AND a.is_active = true
    ) INTO has_permission;
    
    -- إذا لم توجد صلاحية عبر الدور، تحقق من المجموعات
    IF NOT has_permission THEN
        SELECT EXISTS (
            SELECT 1 
            FROM public.admin_group_members agm
            JOIN public.group_permissions gp ON agm.group_id = gp.group_id
            JOIN public.permissions p ON gp.permission_id = p.id
            WHERE agm.admin_id = admin_id AND p.code = permission_code
        ) INTO has_permission;
    END IF;
    
    -- إذا كان المسؤول له صلاحيات إضافية في الحقل permissions
    IF NOT has_permission THEN
        SELECT EXISTS (
            SELECT 1 
            FROM public.admins
            WHERE id = admin_id 
              AND is_active = true
              AND permissions ? permission_code
              AND (permissions->>permission_code)::boolean = true
        ) INTO has_permission;
    END IF;
    
    RETURN has_permission;
END;
$$ LANGUAGE plpgsql;
```

## استخدام نموذج التصميم في واجهة المستخدم

عند إنشاء واجهة مستخدم لإدارة الأدوار والصلاحيات:

1. **عرض قائمة الأدوار** مع وصفها ومستواها
2. **عرض قائمة الصلاحيات المتاحة** مقسمة حسب الموارد والإجراءات
3. **واجهة لإدارة صلاحيات كل دور** تتيح إضافة/إزالة الصلاحيات
4. عند حفظ التغييرات، يتم:
   - **إضافة سجلات جديدة** في جدول `role_permissions` للصلاحيات الجديدة
   - **حذف السجلات الموجودة** من جدول `role_permissions` للصلاحيات المزالة

## الخلاصة

استخدام جدول وسيط `role_permissions` لربط الأدوار بالصلاحيات يوفر نظامًا متكاملاً ومرنًا لإدارة الصلاحيات مع الحفاظ على سلامة البيانات وأداء الاستعلامات. 

هذا النهج العلائقي يتفوق على استخدام حقل JSONB لتخزين الصلاحيات من حيث:
- سلامة البيانات
- أداء الاستعلامات
- سهولة الإدارة والصيانة
- قابلية التوسع

مع ذلك، يمكن الاحتفاظ بحقل JSONB كطبقة إضافية للصلاحيات الخاصة أو المؤقتة أو للتوافق مع الأنظمة السابقة.
