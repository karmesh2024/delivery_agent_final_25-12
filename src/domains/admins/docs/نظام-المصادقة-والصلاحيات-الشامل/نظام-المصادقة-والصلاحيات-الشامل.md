## 1. نظرة عامة على النظام

يستخدم النظام **Supabase** كخدمة للمصادقة وقاعدة البيانات، مع تكامل مع Redux لإدارة حالة المصادقة في التطبيق. النظام يتكون من المكونات الرئيسية التالية:

- **نظام مصادقة Supabase**: للتعامل مع عمليات تسجيل الدخول وإدارة الجلسات
- **جدول المسؤولين (admins)**: لتخزين معلومات المستخدمين المسؤولين
- **نظام الأدوار والصلاحيات**: للتحكم في وصول المستخدمين إلى ميزات مختلفة
- **المحفزات (Triggers)**: لربط حسابات المستخدمين في Supabase Auth مع جدول المسؤولين
- **Row Level Security (RLS)**: لتأمين الوصول إلى البيانات على مستوى الصف

## 2. بنية قاعدة البيانات

### الجداول الرئيسية

1. **جدول auth.users (مدار بواسطة Supabase Auth)**
   - معلومات المستخدم الأساسية (البريد الإلكتروني، كلمة المرور المشفرة، إلخ)

2. **جدول public.admins**
   ```sql
   CREATE TABLE public.admins (
     id uuid NOT NULL DEFAULT extensions.uuid_generate_v4(),
     user_id uuid REFERENCES auth.users(id) ON DELETE CASCADE,
     name text NULL,
     is_active boolean NOT NULL DEFAULT true,
     email text NOT NULL,
     role_id uuid REFERENCES public.roles(id),
     permissions jsonb NULL,
     created_at timestamptz NOT NULL DEFAULT CURRENT_TIMESTAMP,
     updated_at timestamptz NULL,
     last_login timestamptz NULL,
     CONSTRAINT admins_pkey PRIMARY KEY (id)
   );
   ```

3. **جدول public.roles**
   ```sql
   CREATE TABLE public.roles (
     id uuid NOT NULL DEFAULT extensions.uuid_generate_v4(),
     name text NOT NULL,
     description text NULL,
     permissions jsonb NOT NULL,
     is_system boolean NOT NULL DEFAULT false,
     created_at timestamptz NOT NULL DEFAULT CURRENT_TIMESTAMP,
     updated_at timestamptz NULL,
     CONSTRAINT roles_pkey PRIMARY KEY (id)
   );
   ```

4. **جدول public.permissions**
   ```sql
   CREATE TABLE public.permissions (
     id uuid NOT NULL DEFAULT extensions.uuid_generate_v4(),
     code text NOT NULL UNIQUE,
     name text NOT NULL,
     description text NULL,
     group_name text NULL,
     created_at timestamptz NOT NULL DEFAULT CURRENT_TIMESTAMP,
     CONSTRAINT permissions_pkey PRIMARY KEY (id)
   );
   ```

### العلاقات بين الجداول

- كل مستخدم في `auth.users` يمكن أن يكون له سجل واحد في `public.admins`
- كل مسؤول في `public.admins` ينتمي إلى دور واحد في `public.roles`
- كل دور في `public.roles` يحتوي على مجموعة من الصلاحيات مخزنة في حقل `permissions` كـ JSON
- جدول `public.permissions` يحتوي على تعريفات الصلاحيات المتاحة في النظام

### المحفزات (Triggers) والوظائف (Functions)

تم إنشاء محفزات في قاعدة البيانات لربط نظام مصادقة Supabase مع جدول `admins`:

```sql
-- وظيفة لإضافة مستخدم جديد إلى جدول المسؤولين عند إنشاء حساب جديد
CREATE OR REPLACE FUNCTION public.process_new_admin_user()
RETURNS TRIGGER AS $$
DECLARE
  default_role_id uuid;
  admins_table_exists boolean;
BEGIN
  -- التحقق من وجود جدول المسؤولين
  SELECT EXISTS (
    SELECT FROM information_schema.tables 
    WHERE table_schema = 'public' AND table_name = 'admins'
  ) INTO admins_table_exists;
  
  IF NOT admins_table_exists THEN
    RETURN NEW; -- المستخدم تم إنشاؤه في auth.users لكن لا يوجد جدول admins
  END IF;

  -- الحصول على الدور الافتراضي (عادة 'viewer')
  SELECT id INTO default_role_id FROM public.roles WHERE name = 'viewer' LIMIT 1;
  
  -- إنشاء سجل جديد في جدول المسؤولين
  INSERT INTO public.admins (user_id, email, role_id)
  VALUES (NEW.id, NEW.email, default_role_id);
  
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

-- محفز عند إنشاء مستخدم جديد
CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION public.process_new_admin_user();

-- محفز عند تحديث مستخدم
CREATE TRIGGER on_auth_user_updated
  AFTER UPDATE ON auth.users
  FOR EACH ROW EXECUTE FUNCTION public.update_admin_on_user_change();
```

### وظائف RPC للتحقق من الصلاحيات

```sql
-- وظيفة محسنة للتحقق من صلاحيات المستخدم
CREATE OR REPLACE FUNCTION check_admin_permission_enhanced(
  admin_id uuid,
  permission_code text
) RETURNS boolean AS $$
DECLARE
  has_permission boolean;
  admin_record record;
  role_permissions jsonb;
  admin_permissions jsonb;
BEGIN
  -- جلب بيانات المسؤول
  SELECT a.*, r.permissions AS role_perms
  INTO admin_record
  FROM public.admins a
  LEFT JOIN public.roles r ON a.role_id = r.id
  WHERE a.id = admin_id;
  
  IF admin_record IS NULL THEN
    RETURN false;
  END IF;
  
  -- تحديد صلاحيات الدور
  role_permissions := admin_record.role_perms;
  
  -- تحديد الصلاحيات المخصصة للمسؤول (إن وجدت)
  admin_permissions := admin_record.permissions;
  
  -- التحقق من وجود الصلاحية
  -- أولاً في الصلاحيات المخصصة للمسؤول
  IF admin_permissions IS NOT NULL AND admin_permissions ? permission_code THEN
    has_permission := (admin_permissions ->> permission_code)::boolean;
  -- ثم في صلاحيات الدور
  ELSIF role_permissions IS NOT NULL AND role_permissions ? permission_code THEN
    has_permission := (role_permissions ->> permission_code)::boolean;
  ELSE
    has_permission := false;
  END IF;
  
  RETURN has_permission;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

## 3. آلية تسجيل المستخدمين الجدد

تتم عملية تسجيل المستخدمين الجدد عبر الخطوات التالية:

1. **واجهة التسجيل**: المستخدم يملأ نموذج التسجيل (البريد الإلكتروني، كلمة المرور)
2. **طلب التسجيل**: يتم إرسال طلب إلى `authApi.register` الذي يتعامل مع Supabase Auth
3. **إنشاء المستخدم**: يتم إنشاء المستخدم في جدول `auth.users`
4. **تشغيل المحفز**: يتم تشغيل محفز `on_auth_user_created` الذي ينشئ سجلاً في جدول `admins`
5. **تأكيد البريد الإلكتروني**: يتم إرسال بريد إلكتروني للتأكيد (حسب الإعدادات)
6. **تعيين الدور**: يتم تعيين دور افتراضي (عادة "viewer") للمستخدم الجديد

### شيفرة التسجيل الرئيسية

```typescript
// في ملف authApi.ts
register: async (registration: AdminRegistration): Promise<AuthResult> => {
  try {
    // إنشاء حساب المستخدم في Supabase
    const { data: authData, error: authError } = await supabase.auth.signUp({
      email: registration.email,
      password: registration.password,
    });

    if (authError) throw authError;

    return {
      success: true,
      token: authData.session?.access_token,
      admin: undefined, // لن يكون هناك مسؤول حتى يتم تأكيد البريد الإلكتروني
      emailConfirmationRequired: authData.user ? !authData.user.email_confirmed_at : true,
      email: registration.email
    };
  } catch (error) {
    console.error('Registration error:', error);
    return {
      success: false,
      message: error instanceof Error ? error.message : 'حدث خطأ أثناء التسجيل'
    };
  }
}
```

## 4. آلية تسجيل الدخول

تتم عملية تسجيل الدخول عبر الخطوات التالية:

1. **واجهة تسجيل الدخول**: المستخدم يدخل بريده الإلكتروني وكلمة المرور
2. **طلب تسجيل الدخول**: يتم إرسال طلب إلى `authApi.login`
3. **مصادقة Supabase**: تتم مصادقة بيانات الاعتماد عبر Supabase Auth
4. **التحقق من المسؤول**: يتم التحقق من وجود المستخدم في جدول `admins` وأنه نشط
5. **جلب بيانات المسؤول**: يتم جلب بيانات المسؤول بما في ذلك الدور والصلاحيات
6. **تخزين حالة المصادقة**: يتم تخزين حالة المصادقة في Redux

### شيفرة تسجيل الدخول الرئيسية

```typescript
// في ملف authApi.ts
login: async (credentials: AdminCredentials): Promise<AuthResult> => {
  try {
    // تسجيل الدخول باستخدام Supabase
    const { data: authData, error: authError } = await supabase.auth.signInWithPassword({
      email: credentials.email,
      password: credentials.password,
    });

    if (authError) throw authError;

    // التحقق من وجود المستخدم في جدول المسؤولين
    const { data: adminData, error: adminError } = await supabase
      .from('admins')
      .select(`
        id, email, name, role_id, permissions,
        role:roles (id, name, permissions)
      `)
      .eq('user_id', authData.user.id)
      .eq('is_active', true)
      .single();

    if (adminError || !adminData) {
      // إذا لم يتم العثور على المسؤول أو حدث خطأ
      await supabase.auth.signOut();
      throw new Error('غير مصرح لهذا المستخدم بالوصول إلى لوحة التحكم.');
    }

    // إنشاء كائن المسؤول
    const admin: Admin = {
      id: adminData.id,
      email: adminData.email,
      name: adminData.name || '',
      role: adminData.role as AdminRole,
      permissions: adminData.permissions || {},
    };

    // تحديث وقت آخر تسجيل دخول
    await supabase
      .from('admins')
      .update({ last_login: new Date().toISOString() })
      .eq('id', admin.id);

    return {
      success: true,
      admin,
      token: authData.session?.access_token
    };
  } catch (error) {
    console.error('Login error:', error);
    return {
      success: false,
      message: error instanceof Error ? error.message : 'حدث خطأ أثناء تسجيل الدخول'
    };
  }
}
```

### التحقق من الصلاحيات

يتم التحقق من صلاحيات المستخدم عبر دالة `hasPermission`:

```typescript
// في ملف permissions.ts
export async function hasPermission(admin: Admin | null | undefined, permissionCode: string): Promise<boolean> {
  if (!admin) return false;
  
  // التحقق من الصلاحيات المخصصة للمسؤول
  if (admin.permissions && typeof admin.permissions[permissionCode] === 'boolean') {
    return admin.permissions[permissionCode] === true;
  }
  
  // التحقق من صلاحيات الدور
  if (admin.role && admin.role.permissions && typeof admin.role.permissions[permissionCode] === 'boolean') {
    return admin.role.permissions[permissionCode] === true;
  }
  
  // استخدام دالة RPC محسنة للتحقق من الصلاحيات
  try {
    const { data, error } = await supabase.rpc('check_admin_permission_enhanced', {
      admin_id: admin.id,
      permission_code: permissionCode
    });
    
    if (error) throw error;
    return !!data;
  } catch (error) {
    console.error('Error checking permission:', error);
    return false;
  }
}
```

## 6. الحماية وأمان الوصول

يتم تأمين الوصول إلى البيانات والصفحات عبر عدة طبقات:

### 1. حماية الصفحات

يتم استخدام مكون `AuthProvider` للتحقق من المصادقة وتوجيه المستخدمين غير المصادقين إلى صفحة تسجيل الدخول:

```tsx
// في ملف AuthProvider.tsx
export function AuthProvider({ children }: { children: React.ReactNode }) {
  const router = useRouter();
  const pathname = usePathname();
  const dispatch = useAppDispatch();
  const { isAuthenticated, loading } = useAppSelector((state) => state.auth);
  
  useEffect(() => {
    // التحقق من المصادقة
    dispatch(checkAuth());
  }, [dispatch]);
  
  useEffect(() => {
    // قائمة المسارات العامة
    const publicPaths = ['/login', '/register', '/forgot-password', '/reset-password'];
    const isPublicPath = publicPaths.some(path => pathname?.startsWith(path));
    
    // إذا كان المستخدم غير مصادق ويحاول الوصول إلى مسار محمي
    if (!isPublicPath && !loading && !isAuthenticated) {
      const returnUrl = encodeURIComponent(pathname || '/');
      router.push(`/login?returnUrl=${returnUrl}`);
    }
    
    // إذا كان المستخدم مصادقًا ويحاول الوصول إلى صفحة تسجيل الدخول أو التسجيل
    if (isPublicPath && isAuthenticated) {
      router.push('/');
    }
  }, [isAuthenticated, loading, pathname, router]);
  
  return children;
}
```

### 2. حماية المكونات

يتم استخدام مكون `PermissionGuard` للتحقق من الصلاحيات قبل عرض مكونات معينة:

```tsx
// في ملف PermissionGuard.tsx
export function PermissionGuard({ permissionCode, children, fallback = null }: PermissionGuardProps) {
  const { currentAdmin } = useAppSelector((state) => state.auth);
  const [hasAccess, setHasAccess] = useState(false);
  
  useEffect(() => {
    async function checkPermission() {
      const result = await hasPermission(currentAdmin, permissionCode);
      setHasAccess(result);
    }
    
    checkPermission();
  }, [currentAdmin, permissionCode]);
  
  return hasAccess ? children : fallback;
}

// استخدام المكون

  <CreateOrderButton />
</PermissionGuard>
```

### 3. أمان قاعدة البيانات (RLS)

يتم استخدام Row Level Security لتأمين الوصول إلى البيانات على مستوى قاعدة البيانات:

```sql
-- مثال على سياسة RLS لجدول المسؤولين
CREATE POLICY admins_policy
  ON public.admins
  FOR ALL
  TO authenticated
  USING (user_id = auth.uid() OR EXISTS (
    SELECT 1 FROM public.admins a
    JOIN public.roles r ON a.role_id = r.id
    WHERE a.user_id = auth.uid()
    AND r.permissions->>'admins:manage' = 'true'
  ));
```

## 7. تكامل النظام مع Redux

يتم استخدام Redux لإدارة حالة المصادقة في التطبيق عبر `authSlice`:

```typescript
// في ملف authSlice.ts
export const authSlice = createSlice({
  name: 'auth',
  initialState,
  reducers: {
    // تحديد المسؤول الحالي
    setCurrentAdmin: (state, action: PayloadAction<Admin | null>) => {
      state.currentAdmin = action.payload;
      state.isAuthenticated = !!action.payload;
    },
    // مسح الأخطاء
    clearError: (state) => {
      state.error = null;
    },
    // تحديث صلاحيات المسؤول
    updateAdminPermissions: (state, action: PayloadAction<Record<string, boolean>>) => {
      if (state.currentAdmin) {
        state.currentAdmin.permissions = {
          ...state.currentAdmin.permissions,
          ...action.payload
        };
      }
    }
  },
  extraReducers: (builder) => {
    builder
      // معالجة تسجيل الدخول
      .addCase(login.pending, (state) => {
        state.loading = true;
        state.error = null;
      })
      .addCase(login.fulfilled, (state, action) => {
        state.loading = false;
        if (action.payload.success && action.payload.admin) {
          state.isAuthenticated = true;
          state.currentAdmin = action.payload.admin;
          state.token = action.payload.token || null;
        } else {
          state.error = action.payload.message || 'حدث خطأ أثناء تسجيل الدخول';
        }
      })
      // ... معالجة بقية الأحداث
  }
});
```

## 8. مخططات سير العمل

### عملية تسجيل الدخول

```
المستخدم -> يدخل بيانات الاعتماد -> authApi.login -> Supabase Auth -> التحقق في جدول admins -> تخزين الحالة في Redux -> توجيه إلى الصفحة الرئيسية
```

### عملية التسجيل

```
المستخدم -> يملأ نموذج التسجيل -> authApi.register -> Supabase Auth -> تشغيل محفز -> إنشاء سجل في جدول admins -> إرسال بريد تأكيد -> توجيه إلى صفحة تأكيد
```

### عملية التحقق من الصلاحيات

```
زيارة صفحة محمية -> AuthProvider -> التحقق من المصادقة -> التحقق من الصلاحيات -> عرض المحتوى أو توجيه إلى صفحة غير مصرح
```

## 10. خطوات التطوير المستقبلية

1. **تحسين إدارة الأدوار**
   - إضافة واجهة رسومية لإدارة الأدوار وتخصيص الصلاحيات
   - دعم تعريف أدوار مخصصة

2. **تعزيز أمان النظام**
   - إضافة المصادقة الثنائية (2FA)
   - تحسين تخزين وإدارة الجلسات
   - إضافة تسجيل لمحاولات الدخول الفاشلة

3. **تحسين تجربة المستخدم**
   - تذكر جلسة المستخدم لفترة أطول
   - تحسين واجهات تسجيل الدخول والتسجيل
   - إضافة خيار "نسيت كلمة المرور"

4. **توسيع نطاق الصلاحيات**
   - دعم صلاحيات متعددة المستويات
   - إضافة صلاحيات محددة بالكيان (مثلاً، إدارة مندوبين محددين فقط)
   - تحسين أداء التحقق من الصلاحيات

5. **تكامل أفضل مع بقية أجزاء التطبيق**
   - ربط الصلاحيات بعناصر واجهة المستخدم بشكل ديناميكي
   - تخصيص لوحة التحكم بناءً على صلاحيات المستخدم
