# توثيق تفصيلي مقترح لنظام إدارة المدفوعات والمحافظ المالية

## 1. مقدمة

يهدف هذا التوثيق إلى تقديم تصميم تفصيلي لقاعدة البيانات والوظائف الأساسية اللازمة لإنشاء نظام متكامل لإدارة المدفوعات والمحافظ المالية عبر تطبيقات العملاء، المناديب، والمسؤولين (الأدمن). يعتمد هذا التصميم على استخدام Supabase (PostgreSQL) كقاعدة بيانات.

## 2. نظرة عامة على سير العمل والكيانات الأساسية

النظام يتمحور حول مفهوم "المحفظة الإلكترونية" لكل مستخدم (عميل، مندوب، وكيل/شركة). يمكن للمستخدمين شحن محافظهم، والدفع للطلبات من خلالها، واستقبال الأموال فيها. يمكن للمناديب والوكلاء طلب سحب أرصدتهم إلى طرق دفع خارجية. يدير الأدمن النظام بأكمله، بما في ذلك الموافقة على عمليات السحب، تسوية الحسابات، ومراقبة المعاملات.

**الكيانات الرئيسية:**

*   **المستخدمون (`users`):** هم الأفراد أو الكيانات (عملاء، مناديب، وكلاء، شركات، مسؤولون) الذين يتفاعلون مع النظام. سيتم الاعتماد على جدول `auth.users` الموجود في Supabase.
*   **المحافظ (`wallets`):** الحسابات الافتراضية التي تحتفظ برصيد المستخدمين.
*   **معاملات المحفظة (`wallet_transactions`):** سجل لكل حركة مالية تتم على أي محفظة.
*   **طرق الدفع المتاحة (`payment_methods`):** قائمة بأنواع طرق الدفع التي يدعمها النظام (مثل فودافون كاش، تحويل بنكي).
*   **طرق الدفع الخاصة بالمستخدم (`user_payment_methods`):** تفاصيل طرق الدفع التي قام المستخدم بإضافتها لحسابه (مثل رقم فودافون كاش معين).
*   **الطلبات (`orders`):** سيتم تحديث جدول الطلبات الحالي ليشمل معلومات متعلقة بالدفع.
*   **طلبات السحب/التسويات (`payouts`):** لعمليات سحب الأرصدة من قبل المناديب والوكلاء.

## 4. دوال قاعدة البيانات المقترحة (PostgreSQL Functions)

يمكن إنشاء دوال في PostgreSQL لتنفيذ منطق معقد أو عمليات متكررة بشكل آمن وفعال (Transactional).

**أ. دالة لمعالجة معاملة محفظة (إضافة أو خصم وتحديث الرصيد):**
   هذه دالة حرجة لضمان سلامة البيانات (Atomicity).

```sql
CREATE OR REPLACE FUNCTION public.process_wallet_transaction(
    p_wallet_id uuid,
    p_amount NUMERIC, -- موجب للإضافة، سالب للخصم (أو استخدم عمود اتجاه منفصل)
    p_transaction_type transaction_type_enum,
    p_description TEXT DEFAULT NULL,
    p_related_order_id uuid DEFAULT NULL,
    p_related_user_payment_method_id uuid DEFAULT NULL,
    p_related_payout_id uuid DEFAULT NULL,
    p_external_transaction_id VARCHAR DEFAULT NULL,
    p_initiated_by_user_id uuid DEFAULT NULL,
    p_metadata JSONB DEFAULT NULL
)
RETURNS public.wallet_transactions -- ترجع المعاملة التي تم إنشاؤها
LANGUAGE plpgsql
SECURITY DEFINER -- هام: لتجاوز RLS إذا لزم الأمر داخل الدالة بعد التحقق من الصلاحيات
AS $$
DECLARE
    v_wallet public.wallets;
    v_balance_before NUMERIC;
    v_balance_after NUMERIC;
    v_transaction_record public.wallet_transactions;
BEGIN
    -- الحصول على المحفظة وتأمين الصف للتحديث
    SELECT * INTO v_wallet FROM public.wallets WHERE id = p_wallet_id FOR UPDATE;

    IF NOT FOUND THEN
        RAISE EXCEPTION 'Wallet not found: %', p_wallet_id;
    END IF;

    IF v_wallet.status != 'ACTIVE' THEN
        RAISE EXCEPTION 'Wallet is not active: %', p_wallet_id;
    END IF;

    v_balance_before := v_wallet.balance;

    -- التحقق من الرصيد الكافي إذا كانت معاملة خصم (p_amount سالب)
    IF p_amount < 0 AND (v_wallet.balance + p_amount) < 0 THEN
        RAISE EXCEPTION 'Insufficient balance in wallet: %', p_wallet_id;
    END IF;

    v_balance_after := v_wallet.balance + p_amount;

    -- تحديث رصيد المحفظة
    UPDATE public.wallets
    SET balance = v_balance_after, updated_at = NOW()
    WHERE id = p_wallet_id;

    -- إنشاء سجل معاملة
    INSERT INTO public.wallet_transactions (
        wallet_id, transaction_type, amount, currency, status,
        description, related_order_id, related_user_payment_method_id, related_payout_id,
        external_transaction_id, balance_before, balance_after,
        initiated_by_user_id, metadata
    )
    VALUES (
        p_wallet_id, p_transaction_type, ABS(p_amount), v_wallet.currency, 'COMPLETED', -- افترض أن المعاملة تكتمل فورًا
        p_description, p_related_order_id, p_related_user_payment_method_id, p_related_payout_id,
        p_external_transaction_id, v_balance_before, v_balance_after,
        p_initiated_by_user_id, p_metadata
    )
    RETURNING * INTO v_transaction_record;

    RETURN v_transaction_record;

EXCEPTION
    WHEN OTHERS THEN
        -- يمكنك هنا تسجيل الخطأ أو اتخاذ إجراءات أخرى
        RAISE; -- إعادة إظهار الخطأ ليتم التعامل معه من قبل المستدعي
END;
$$;
```
**ملاحظات على الدالة `process_wallet_transaction`:**
*   **الأمان (`SECURITY DEFINER`):** استخدم بحذر. تأكد من أن صلاحيات استدعاء هذه الدالة محكومة بشكل جيد.
*   **معالجة الأخطاء:** يمكن تحسينها لتسجيل الأخطاء بشكل أفضل.
*   **حالة المعاملة:** حاليًا يتم تعيينها `COMPLETED` مباشرة. قد تحتاج إلى منطق أكثر تعقيدًا لحالات `PENDING` ثم تحديثها لاحقًا.
*   **اتجاه المبلغ:** الدالة الحالية تفترض أن `p_amount` يكون سالبًا للخصم وموجبًا للإضافة. `ABS(p_amount)` يُستخدم لتسجيل القيمة المطلقة في `wallet_transactions.amount`، ويعتمد النوع (`transaction_type`) على تحديد ما إذا كان إيداعًا أو سحبًا.

**ب. دوال Triggers لتحديث `updated_at`:**
   يمكن إنشاء دوال trigger بسيطة لتحديث عمود `updated_at` تلقائيًا عند تعديل أي صف في الجداول.

```sql
CREATE OR REPLACE FUNCTION public.trigger_set_timestamp()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- تطبيق الـ Trigger على الجداول (مثال لجدول wallets)
CREATE TRIGGER set_timestamp_wallets
BEFORE UPDATE ON public.wallets
FOR EACH ROW
EXECUTE FUNCTION public.trigger_set_timestamp();

-- كرر إنشاء الـ Trigger لبقية الجداول: wallet_transactions, user_payment_methods, payouts, payment_methods
```

## 5. سير عمل مقترح لبعض العمليات الرئيسية

### أ. عملية دفع طلب من محفظة العميل:
1.  العميل يختار الدفع من المحفظة.
2.  التطبيق يتحقق من رصيد محفظة العميل.
3.  إذا كان الرصيد كافيًا:
    *   يتم استدعاء الدالة `process_wallet_transaction` لخصم المبلغ من محفظة العميل (`wallet_type = 'CUSTOMER_HOME'`).
        *   `p_amount` يكون سالبًا.
        *   `p_transaction_type = 'ORDER_PAYMENT'`.
        *   `p_related_order_id` هو معرف الطلب.
    *   يتم استدعاء الدالة `process_wallet_transaction` لإضافة المبلغ إلى محفظة إيرادات النظام (`wallet_type = 'SYSTEM_REVENUE'`) أو محفظة الوكيل إذا كان الطلب عبر وكيل.
        *   `p_amount` يكون موجبًا.
        *   `p_transaction_type = 'TRANSFER_IN'` (أو نوع مخصص).
    *   تحديث حالة الطلب (`orders.payment_status = 'PAID'`).
    *   ربط معاملة المحفظة بالطلب (`orders.wallet_transaction_id`).

### ب. شحن محفظة العميل (مثلاً عبر فودافون كاش):
1.  العميل يختار شحن المحفظة ويحدد المبلغ وطريقة الدفع (فودافون كاش).
2.  التطبيق يتكامل مع بوابة الدفع (API فودافون كاش).
3.  عند تأكيد الدفع من البوابة:
    *   يتم إنشاء سجل في `wallet_transactions` بحالة `PENDING` أو `REQUIRES_ACTION`.
    *   بعد التأكيد النهائي (مثلاً OTP أو callback من البوابة):
        *   يتم استدعاء الدالة `process_wallet_transaction` لإضافة المبلغ إلى محفظة العميل.
            *   `p_amount` يكون موجبًا.
            *   `p_transaction_type = 'DEPOSIT'`.
            *   `p_external_transaction_id` هو معرف المعاملة من فودافون كاش.
        *   تحديث حالة المعاملة في `wallet_transactions` إلى `COMPLETED`.

### ج. المندوب يسلم الطلب (الدفع نقدًا من العميل):
1.  المندوب يؤكد استلام المبلغ النقدي من العميل عبر تطبيقه.
2.  خياران للمعالجة:
    *   **الخيار 1 (الأبسط للمندوب):** المبلغ يذهب مباشرة إلى محفظة النظام أو الوكيل. لا يمر عبر محفظة المندوب أولاً.
        *   يتم تسجيل معاملة `DEPOSIT` إلى محفظة النظام (`SYSTEM_FLOAT` أو `SYSTEM_REVENUE`) أو محفظة الوكيل.
    *   **الخيار 2 (المندوب وسيط):**
        *   `process_wallet_transaction`: إيداع في محفظة المندوب (`DELIVERY_BOY`) كـ `DEPOSIT` أو `DELIVERY_PAYMENT_COLLECTION`.
        *   لاحقًا، المندوب يقوم بتسوية هذا المبلغ مع الإدارة (قد يكون تحويل دوري أو يدوي من محفظته إلى محفظة النظام).
3.  تحديث حالة الطلب (`orders.payment_status = 'PAID'`).

### د. المندوب يطلب سحب رصيد من محفظته:
1.  المندوب يطلب سحب مبلغ معين إلى طريقة دفع محفوظة (مثلاً حسابه البنكي).
2.  يتم إنشاء سجل في جدول `payouts` بحالة `PENDING_APPROVAL`.
3.  الأدمن يراجع الطلب في تطبيق الأدمن:
    *   **إذا وافق:**
        *   تحديث حالة `payouts` إلى `APPROVED` ثم `PROCESSING`.
        *   يتم استدعاء الدالة `process_wallet_transaction` لخصم `amount_approved` من محفظة المندوب.
            *   `p_transaction_type = 'PAYOUT_TO_USER'`.
            *   `p_related_payout_id` هو معرف طلب السحب.
        *   الأدمن يقوم بتحويل المبلغ يدويًا أو عبر تكامل مع نظام دفع.
        *   عند إتمام التحويل، يتم تحديث حالة `payouts` إلى `COMPLETED` وإضافة `transaction_reference`.
    *   **إذا رفض:**
        *   تحديث حالة `payouts` إلى `REJECTED` مع ذكر السبب.

## 6. اعتبارات إضافية

*   **الأمان (RLS - Row Level Security):** استخدم RLS في Supabase بفعالية لضمان أن المستخدمين يمكنهم فقط الوصول إلى محافظهم ومعاملاتهم وطرق الدفع الخاصة بهم. الأدمن سيكون له صلاحيات أوسع.
*   **التحقق من صحة البيانات (Validation):** قم بإجراء تحقق شامل من صحة البيانات على مستوى التطبيق وقاعدة البيانات.
*   **التسجيل (Logging):** سجل جميع العمليات الهامة والوصول إلى البيانات الحساسة.
*   **الاختبار:** قم باختبار جميع سيناريوهات الدفع والسحب بدقة.
*   **المعاملات (Transactions):** تأكد من أن العمليات التي تتضمن خطوات متعددة (مثل خصم من محفظة وإضافة لأخرى) تتم داخل معاملة قاعدة بيانات واحدة لضمان الاتساق (Atomicity). الدالة `process_wallet_transaction` تساعد في ذلك.
*   **العملات:** التصميم الحالي يدعم عملة افتراضية واحدة (EGP). إذا كنت بحاجة لدعم عملات متعددة، ستحتاج إلى تعديلات إضافية (مثل جداول لأسعار الصرف، وتحديد عملة كل معاملة بشكل دقيق).

هذا التوثيق يوفر أساسًا قويًا. ستحتاج إلى تكييفه وتوسيعه بناءً على المتطلبات الدقيقة والتفاصيل الخاصة بتطبيقك.

## سياسات الأمان وإدارة الصلاحيات (RLS Policies)

### دوال التحقق من الصلاحيات

تم إنشاء دالة مخصصة للتحقق من صلاحيات المشرفين للتعامل مع المحافظ والمدفوعات:

```sql
CREATE OR REPLACE FUNCTION public.check_wallet_permission()
RETURNS BOOLEAN
LANGUAGE plpgsql
SECURITY DEFINER
AS $$
BEGIN
  RETURN EXISTS (
    SELECT 1 
    FROM admins a
    WHERE a.user_id = auth.uid()
    AND a.is_active = true
    AND (
      (a.permissions->>'wallets:manage')::boolean = true
      OR check_admin_permission_enhanced(a.id, 'wallets:manage')
    )
  );
END;
$$;
```

هذه الدالة:
- تتحقق من أن المستخدم مشرف نشط
- تتحقق من امتلاك المشرف لصلاحية `wallets:manage`
- تستخدم نظام الصلاحيات الموجود في جدول `admins`

### سياسات المحافظ (Wallets)

1. **عرض المحافظ**:
   ```sql
   CREATE POLICY "Users can view their own wallets"
   ON public.wallets FOR SELECT
   USING (auth.uid() = user_id OR public.check_wallet_permission());
   ```
   - يمكن للمستخدمين رؤية محافظهم فقط
   - يمكن للمشرفين المصرح لهم رؤية جميع المحافظ

2. **إنشاء المحافظ**:
   ```sql
   CREATE POLICY "System can create user wallets"
   ON public.wallets FOR INSERT
   WITH CHECK (auth.uid() = user_id OR public.check_wallet_permission());
   ```
   - يمكن للمستخدمين إنشاء محافظ لأنفسهم فقط
   - يمكن للنظام والمشرفين المصرح لهم إنشاء محافظ للمستخدمين

3. **إدارة المحافظ**:
   ```sql
   CREATE POLICY "Admins can manage wallets"
   ON public.wallets FOR ALL
   USING (public.check_wallet_permission())
   WITH CHECK (public.check_wallet_permission());
   ```
   - يمكن للمشرفين المصرح لهم إدارة جميع المحافظ (تحديث، حذف، إلخ)

### سياسات معاملات المحافظ (Wallet Transactions)

1. **عرض المعاملات**:
   ```sql
   CREATE POLICY "Users can view their wallet transactions"
   ON public.wallet_transactions FOR SELECT
   USING (EXISTS (
     SELECT 1 FROM public.wallets w
     WHERE w.id = wallet_id AND w.user_id = auth.uid()
   ) OR public.check_wallet_permission());
   ```
   - يمكن للمستخدمين رؤية معاملات محافظهم فقط
   - يمكن للمشرفين المصرح لهم رؤية جميع المعاملات

2. **إنشاء المعاملات**:
   ```sql
   CREATE POLICY "System can create wallet transactions"
   ON public.wallet_transactions FOR INSERT
   WITH CHECK (EXISTS (
     SELECT 1 FROM public.wallets w
     WHERE w.id = wallet_id 
     AND (w.user_id = auth.uid() OR public.check_wallet_permission())
   ));
   ```
   - يمكن إنشاء معاملات فقط للمحافظ التي يملكها المستخدم
   - يمكن للنظام والمشرفين المصرح لهم إنشاء معاملات لأي محفظة

### سياسات طرق الدفع (Payment Methods)

1. **عرض وإدارة طرق الدفع**:
   ```sql
   CREATE POLICY "Users can view their payment methods"
   ON public.user_payment_methods FOR SELECT
   USING (auth.uid() = user_id OR public.check_wallet_permission());

   CREATE POLICY "Users can manage their payment methods"
   ON public.user_payment_methods FOR INSERT
   WITH CHECK (auth.uid() = user_id);
   ```
   - يمكن للمستخدمين إدارة طرق الدفع الخاصة بهم فقط
   - يمكن للمشرفين المصرح لهم رؤية وإدارة جميع طرق الدفع

### سياسات طلبات السحب (Payouts)

1. **عرض طلبات السحب**:
   ```sql
   CREATE POLICY "Users can view their payouts"
   ON public.payouts FOR SELECT
   USING (auth.uid() = user_id OR public.check_wallet_permission());
   ```
   - يمكن للمستخدمين رؤية طلبات السحب الخاصة بهم فقط
   - يمكن للمشرفين المصرح لهم رؤية جميع طلبات السحب

2. **إنشاء طلبات السحب**:
   ```sql
   CREATE POLICY "Users can request payouts"
   ON public.payouts FOR INSERT
   WITH CHECK (
     auth.uid() = user_id 
     AND EXISTS (
       SELECT 1 FROM public.wallets w 
       WHERE w.id = wallet_id 
       AND w.user_id = auth.uid()
     )
   );
   ```
   - يمكن للمستخدمين إنشاء طلبات سحب من محافظهم فقط
   - يتم التحقق من ملكية المحفظة قبل السماح بإنشاء طلب السحب

3. **تحديث وإلغاء طلبات السحب**:
   ```sql
   CREATE POLICY "Users can update pending payouts"
   ON public.payouts FOR UPDATE
   USING (auth.uid() = user_id AND status = 'PENDING_APPROVAL');

   CREATE POLICY "Users can cancel pending payouts"
   ON public.payouts FOR DELETE
   USING (auth.uid() = user_id AND status = 'PENDING_APPROVAL');
   ```
   - يمكن للمستخدمين تحديث أو إلغاء طلبات السحب الخاصة بهم فقط
   - يمكن التحديث أو الإلغاء فقط للطلبات في حالة "قيد الانتظار"

### ملاحظات هامة

1. **الأمان**:
   - جميع السياسات تعتمد على نظام الصلاحيات الموجود في جدول `admins`
   - يتم التحقق من أن المشرف نشط (`is_active = true`) قبل منح أي صلاحيات
   - يتم التحقق من ملكية المحفظة في جميع العمليات

2. **المرونة**:
   - يمكن تعديل الصلاحيات من خلال تحديث حقل `permissions` في جدول `admins`
   - يمكن إضافة المزيد من السياسات أو تعديلها حسب احتياجات التطبيق

3. **التكامل**:
   - السياسات تتكامل مع نظام إدارة المشرفين الحالي
   - تدعم السياسات العمليات المختلفة في النظام (العملاء، المناديب، المشرفين)

آمل أن يكون هذا التوثيق التفصيلي مفيدًا كنقطة انطلاق لمشروعك!

## 7. تحديثات نظام المدفوعات في الطلبات

### 7.1. تحديث جدول الطلبات

تم تحديث جدول `customer_orders` لدعم نظام المدفوعات من خلال إضافة الأعمدة التالية:

```sql
ALTER TABLE public.customer_orders
ADD COLUMN payment_status order_payment_status_enum DEFAULT 'PENDING_PAYMENT',
ADD COLUMN payment_method_id uuid REFERENCES public.payment_methods(id),
ADD COLUMN wallet_transaction_id uuid REFERENCES public.wallet_transactions(id),
ADD COLUMN payment_amount NUMERIC(15, 2),
ADD COLUMN payment_currency VARCHAR(3) DEFAULT 'EGP',
ADD COLUMN payment_metadata JSONB;
```

**الأعمدة الجديدة**:
- `payment_status`: حالة الدفع للطلب (معلق، مدفوع، فشل، إلخ)
- `payment_method_id`: طريقة الدفع المستخدمة
- `wallet_transaction_id`: معرف معاملة المحفظة المرتبطة بالطلب
- `payment_amount`: مبلغ الدفع
- `payment_currency`: عملة الدفع
- `payment_metadata`: بيانات إضافية متعلقة بالدفع

### 7.2. سياسات الأمان للطلبات

تم تنفيذ سياسات RLS لضمان الوصول الآمن للطلبات ومدفوعاتها:

1. **سياسة عرض الطلبات للعملاء**:
   ```sql
   CREATE POLICY "Customers can view their own orders"
   ON public.customer_orders
   FOR SELECT
   USING (auth.uid() = customer_id);
   ```
   - يمكن للعملاء رؤية طلباتهم فقط
   - يتم التحقق من معرف المستخدم مقابل `customer_id`

2. **سياسة تحديث الطلبات للمناديب**:
   ```sql
   CREATE POLICY "Delivery agents can update orders"
   ON public.customer_orders
   FOR UPDATE
   USING (auth.uid() = profile_id);
   ```
   - يمكن للمناديب تحديث الطلبات المسندة إليهم فقط
   - يتم التحقق من معرف المستخدم مقابل `profile_id`

3. **سياسة إدارة الطلبات للمشرفين**:
   ```sql
   CREATE POLICY "Admins can manage orders"
   ON public.customer_orders
   FOR ALL
   USING (
       EXISTS (
           SELECT 1 
           FROM admins a
           WHERE a.user_id = auth.uid()
           AND a.is_active = true
           AND (
               (a.permissions->>'orders:manage')::boolean = true
               OR (a.permissions->>'payments:manage')::boolean = true
               OR check_admin_permission_enhanced(a.id, 'orders:manage')
               OR check_admin_permission_enhanced(a.id, 'payments:manage')
           )
       )
   );
   ```
   - يمكن للمشرفين المصرح لهم إدارة جميع الطلبات
   - يتم التحقق من الصلاحيات المطلوبة (`orders:manage` أو `payments:manage`)
   - يجب أن يكون المشرف نشطاً (`is_active = true`)

### 7.3. تكامل نظام المدفوعات مع الطلبات

يتم تكامل نظام المدفوعات مع الطلبات من خلال:

1. **ربط المعاملات**:
   - كل طلب يمكن أن يرتبط بمعاملة محفظة (`wallet_transaction_id`)
   - يمكن تتبع حالة الدفع من خلال `payment_status`

2. **طرق الدفع**:
   - يمكن تحديد طريقة الدفع لكل طلب (`payment_method_id`)
   - دعم للدفع النقدي والدفع الإلكتروني

3. **المرونة**:
   - حقل `payment_metadata` يسمح بتخزين معلومات إضافية
   - دعم لعملات مختلفة من خلال `payment_currency`
